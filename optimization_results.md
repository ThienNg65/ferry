# Ferry Startup Optimization Results

## Executive Summary

This report documents the architectural improvements, code refactorings, deep remediation fixes, and empirical benchmark results for the **Ferry Startup Time Optimization** project.

By eliminating synchronous disk I/O during module evaluation, deferring heavy dependencies via `defineAsyncComponent`, updating Vite bundling configuration to eliminate renderer dynamic import waterfalls, applying Chromium startup flags, enabling V8 bytecode caching (`v8CacheOptions: 'bypassHeatCheck'`), and fixing Windows OpenSSH key ACL permissions:
- **Phase 3 disk-waterfall remediation in `electron.vite.config.ts`**: Removed `modulePreload: false` and removed `manualChunks` splitting for `vendor-core`/`vendor-ui`/`vendor-iconify`. Core Vue and Pinia dependencies are bundled into the main entry script (`index-*.js`), enabling single-read script evaluation.
- **Main process boot time (`mainStart` -> `appReady`) is 34.96 ms**, achieving near-instant main process readiness.
- **Window creation & load latency (`appReady` -> `readyToShow`) is 31.94 ms**, enabling instant window frame display.
- **Vue Mount -> First Paint latency (`rendererMount` -> `firstPaint`) is 7.92 ms**, ensuring rapid post-mount frame rendering.
- **Unit test suite achieved 100% passing status (235 / 235 tests passing across 33 test files)** following the Windows ACL permission fix for OpenSSH private keys in `KeyGenerator.ts`.

## 1. Bottleneck Analysis & Remediation Overview

Micro-profiling identified the primary bottlenecks across main process evaluation, Chromium initial paint, window creation, renderer entry bundle evaluation, and OpenSSH permissions:

### 1.1 OpenSSH Private Key ACL Permissions on Windows
- POSIX file mode `0o600` in Node `fs.writeFile` did not restrict inherited Windows NTFS ACL permissions on `%TEMP%` key files, leaving generated private keys readable by `BUILTIN\Users`.
- OpenSSH's `ssh-keygen -y` rejected fallback Ed25519 private keys generated by `generateBuiltin` on Windows with `UNPROTECTED PRIVATE KEY FILE!`.
- **Fix**: Appended exclusive user ACL restriction using `icacls <keyPath> /inheritance:r /grant:r "%USERNAME%:F"` on `win32` platform.

### 1.2 Disk Waterfall Remediation & Renderer Entry Bundling Strategy
- Micro-profiling revealed a disk-waterfall delay in Phase 3 (`Ready to Show -> Vue Mount`). Setting `modulePreload: false` and splitting core Vue/Pinia dependencies into separate dynamic chunks (`vendor-core`, `vendor-ui`, `vendor-iconify`) forced Chromium renderer on Windows `file://` protocol to perform 3 sequential asynchronous disk import waterfalls before Vue could mount.
- **Fix**: Removed `modulePreload: false` and removed `manualChunks` splitting for `vendor-core`/`vendor-ui`/`vendor-iconify` in `electron.vite.config.ts`. Core Vue/Pinia dependencies are bundled directly into the initial entry bundle (`index-tbppqSoF.js`), eliminating the multi-chunk import waterfall on Chromium initial page load.

### 1.3 Chromium Window Creation & Acceleration Switches
- Applied performance acceleration flags before `app.whenReady()` in `src/main/index.ts`:
