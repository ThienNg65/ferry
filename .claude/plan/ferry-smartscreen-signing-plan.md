# Fix Windows SmartScreen warning on Ferry's packaged .exe

## Context

Users who download and run Ferry's installer (`Ferry-Setup-<version>.exe`, built via
`electron-builder` and published through [.github/workflows/release.yml](../../.github/workflows/release.yml))
see Windows SmartScreen's "Windows protected your PC" warning on first run. This is not a bug in
Ferry's code — it happens because the installer is **unsigned**. `electron-builder.yml` already
documents this explicitly (lines 19–25): signing only activates if `CSC_LINK` / `CSC_KEY_PASSWORD`
are set, and `forceCodeSigning: false` lets local/CI builds succeed unsigned when they aren't.
Those env vars have never been backed by a real certificate.

There's no way to make SmartScreen stop warning without actually signing the binary with a
certificate that has (or can build) reputation with Microsoft. Paid options (OV cert, EV cert,
Microsoft Trusted Signing) were ruled out — no budget available. The viable no-cost path is
**SignPath.io's free Authenticode signing program for open-source projects**, which several
well-known Electron apps use. This plan sets up everything on the repo side; the account
enrollment step itself is external and must be done by the user in a browser.

## What needs to happen, in order

### 1. Apply to SignPath (user action, outside this repo)

- Go to https://signpath.io/product/open-source and apply as an open-source project.
- Ferry qualifies as: public GitHub repo (`ThienNg65/ferry`), has releases, actively maintained.
  SignPath's vetting looks for genuine usage/maintenance — being upfront about project maturity
  in the application is fine; approval isn't guaranteed and can take some back-and-forth.
- On approval, SignPath's dashboard provides values needed for step 2:
  - Organization ID
  - Project slug
  - Signing policy slug (they provide a `release-signing` and often a `test-signing` policy)
  - Artifact configuration slug
  - An API token (or GitHub OIDC trust setup, which avoids storing a token secret — SignPath
    supports both; prefer OIDC if their current docs support it for GitHub Actions)

**Implementation cannot proceed past this point until those values exist.** Steps 2–4 below
describe the concrete changes to make once they're in hand.

### 2. Change the release workflow to build unsigned, then sign via SignPath, then publish

Currently [.github/workflows/release.yml](../../.github/workflows/release.yml)'s "Package and
publish" step (lines 45–59) does build+sign+publish in one `electron-builder --publish always`
call, with signing gated on `CSC_LINK`/`CSC_KEY_PASSWORD`. That has to split into three stages,
because SignPath signs an already-built artifact rather than plugging into electron-builder's
build step:

1. **Build unsigned**: `npx electron-builder --publish never` — produces `dist\Ferry-Setup-<version>.exe`
   (and `dist\latest.yml`, used by `electron-updater` per [src/main/update/AutoUpdater.ts](../../src/main/update/AutoUpdater.ts)).
2. **Submit for signing**: use SignPath's official GitHub Action (currently
   `signpath/github-action-submit-signing-request`, but confirm the exact action name/inputs
   against SignPath's current docs at setup time — verify before wiring it in) with
   `wait-for-completion: true`, pointing at the unsigned `.exe`, and downloading the signed result
   back over the same path.
3. **Regenerate update metadata, then publish**: the signed `.exe` has different bytes than the
   unsigned one, so `dist\latest.yml`'s `sha512`/`size` fields (generated by electron-builder at
   build time) are now stale — `electron-updater`'s integrity check will reject the signed file
   otherwise. Regenerate `latest.yml`'s hash/size to match the signed exe (electron-builder
   exposes a hash-update helper for this; check `electron-builder`'s docs/CLI for the current
   supported way — e.g. `electron-builder`'s `PublishManager` internals or a small Node script
   using its `hashFile`/`arch`-aware yml writer), then publish both files to the GitHub release —
   either re-invoke electron-builder's publish step against the already-built+resigned artifacts,
   or upload directly via `gh release upload`.

Add the new secrets referenced by the SignPath action (API token, or OIDC config) to the repo's
GitHub Actions secrets — same place `CSC_LINK`/`CSC_KEY_PASSWORD` already live.

### 3. Update electron-builder.yml comments

Update the comment block at lines 19–26 of [electron-builder.yml](../../electron-builder.yml) to
reflect that Windows builds are now signed post-build via SignPath in CI rather than through
electron-builder's native `CSC_LINK` pickup (`forceCodeSigning: false` still stays — local
`npm run package` remains unsigned by design).

### 4. Update docs

- [README.md](../../README.md) lines 77–80 currently describe the `CSC_LINK`/`CSC_KEY_PASSWORD`
  signed-vs-unsigned behavior — update to describe the SignPath-based flow instead.
- [.claude/PROJECT_MAP.md](../PROJECT_MAP.md) lines 105–110 — same update, this is the
  actively-maintained doc other sessions rely on.
- [CHANGELOG.md](../../CHANGELOG.md) — note the switch from "no code-signing certificate" to
  "SignPath-signed Windows builds" once step 2 actually ships.
- [src/main/update/AutoUpdater.ts](../../src/main/update/AutoUpdater.ts) lines 20–25 — its comment
  says auto-update needs "a code-signing certificate, neither of which exists in this dev
  environment"; update once signing is live.

## Verification

- CI-side (steps 2–3) can't be fully verified until real SignPath credentials exist — this is
  vetting-gated, not something to fake with placeholder secrets.
- Once credentials exist: push a version bump to `main`, watch the `release.yml` run in GitHub
  Actions, confirm the "Package and publish" stage completes all three sub-steps and a signed
  `.exe` lands in the GitHub Release.
- Download the signed installer on a real Windows machine and confirm SmartScreen no longer shows
  "Windows protected your PC" (may take a short reputation-building window even for a
  SignPath-signed binary, but should be dramatically faster than an unsigned one).
- Confirm `electron-updater` still accepts the signed build as a valid update target (launch an
  older packaged build, let it check for updates, confirm no hash-mismatch rejection) — this is
  the part most likely to break silently if `latest.yml`'s hash isn't regenerated correctly in
  step 2.3.

## What NOT to do without further confirmation

- Do not submit the SignPath application on the user's behalf (external account/vetting process —
  needs the user's own identity/org details).
- Do not add placeholder/fake secret values to the workflow — the SignPath action names, input
  schema, and secret names above are described from current public knowledge and must be
  double-checked against SignPath's live documentation once the account exists, since CI/CD
  integration details can drift.
