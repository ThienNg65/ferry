# Code Review: Ferry (full project) — Final Combined Report — 2026-07-20

## Summary

This is the merged result of two review rounds: the original pass (SSH/security core,
IPC/secrets boundary, transfer/sync, and two renderer stores) plus a second round of eight
parallel deep-dives covering every remaining subsystem — storage/credentials, terminal/tail/edit
session managers, archive/operations, the full IPC handler surface, every file-browser and shell
UI component, the remaining renderer stores/composables, and build/CI/test-coverage. Every finding
below was verified by reading the actual code (round 2's most impactful new findings — the
`FilePane.vue` keyboard hijack, the `EditSessionManager` leak, the `RemoteShell` abort leak, and the
`tailStreams` cross-session bug — were independently re-confirmed by re-reading the code directly
before inclusion here).

The single worst finding this round is **not** a backend security bug but a renderer one: normal
typing in the file browser (editing the filter box, naming a new folder, editing the breadcrumb
path) can trigger the pane's Delete shortcut and silently delete selected files, because the
keydown handler isn't scoped away from text inputs. Combined with the original path-traversal and
host-key-trust bugs, this brings the running total to **7 critical issues** across the two rounds.

A concurrent MITM against the target (unrelated to the prompting hop) gets silently trusted. | 🔴 Critical |
| 5 | `src/main/edit/EditSessionManager.ts` | 36-45, 65-114, 169-190, 199-216 | `this.edits` is only ever appended to (line 109) and wiped entirely in `disposeAll()` (app-quit only, line 199). `closeAllForSession()` (170-190) stops the watcher/timer but never removes the map entry. Every distinct (session, remotePath) ever edited during the app's lifetime permanently keeps its map entry, its temp directory (containing decrypted remote file content) on disk, and — while the session stays open — a live `FSWatcher`. This both leaks resources for the process lifetime and leaves decrypted remote content sitting on local disk indefinitely with no user-facing way to clear it short of quitting the app. | 🔴 Critical |
| 6 | `src/renderer/src/stores/tailStreams.store.ts` | 27-36 (confirmed: `find((t) => t.remotePath === remotePath)`, no `sessionId` field on `OpenTail`) | `open()` dedups an existing tail tab by `remotePath` alone. Tailing the same path (e.g. `/var/log/syslog`) from a second, different site session reuses the first session's tab instead of starting a new tail — the user silently views the wrong server's log while believing it's the new one. Compounding this, `sessions.store.ts`'s `closeTab()` (confirmed lines 305-326) disposes `remoteFs` and `terminalStreams` for the closing session but never calls into `tailStreams` at all — a closed tab's tail entry lingers forever, and a later tail on an unrelated session can match that stale entry. | 🔴 Critical |
| 7 | `src/main/ssh/RemoteShell.ts` | 104-118 (`execOnce`'s `onAbort`→`fail`→`cleanup`), same pattern at 182-190 (`execLines`) | Confirmed by direct read: aborting a signal passed into `exec()`/`execLines()` only calls `cleanup()` (clears the timer, removes the abort listener) and rejects — it never calls `stream.destroy()`, unlike the timeout path just below it (132-134 / 208-210) which does. `CompressService.compressRemote` and `UnzipService.extractRemote` thread their cancellation `signal` straight into `shell.exec(...)`, so clicking "Cancel" on a remote compress/extract transitions the UI to `cancelled` correctly but leaves the underlying SSH exec channel open until the remote `zip`/`unzip`/`tar` process finishes on its own. ## Test Coverage Gaps

- **Zero test file at all**: `src/main/tail/TailManager.ts`, `src/main/terminal/TerminalManager.ts`, `src/main/monitor/MonitorManager.ts` (its pure `procParse.ts` helpers *are* tested, the manager's tick/stop lifecycle isn't), `src/main/app/AppSettingsStore.ts`, `src/main/bookmarks/BookmarkStore.ts`, `src/main/history/HistoryStore.ts` and `HistoryRecorder.ts`, `src/main/util/concurrency.ts`.
- **All ~20 files under `src/main/ipc/*.ipc.ts`** — no `*.ipc.test.ts` exists for any handler file, despite this being the layer that receives raw, unvalidated renderer input.
- `src/main/fs/LocalFsService.ts` / `RemoteFsService.ts` — no dedicated unit test; only indirectly touched (mocked) inside `EditSessionManager.test.ts`. Symlink resolution, chmod, and batch-delete have no direct test.
- `src/main/ssh/ProxyConnector.ts` — only the HTTP CONNECT path is tested; the SOCKS5 branch (used for hop-0 proxying) has no test, real or mocked.
- `TailManager.sanitizeHistoryLines` / `MonitorManager.sanitizeIntervalMs` — the actual injection-prevention guards for IPC-sourced numeric input — have no dedicated unit test despite being security-relevant.
- `.claude/PROJECT_MAP.md`'s release-pipeline description needs a rewrite to match the current tag-before-publish, no-notes-PATCH shape (see High-Priority Suggestion #6).

## What Looks Good

- **Secrets discipline**: `SiteStore`/`AppSettingsStore` never return plaintext to the renderer (only `hasPassword`/`hasPassphrase` booleans); decryption happens main-process-side, in-memory, at connect time only — verified via grep, no `getDecrypted*` call sites outside `SessionManager`.
- **`BrowserWindow` hardening** matches documentation exactly: `contextIsolation`/`sandbox` true, `nodeIntegration`/`webSecurity` correctly configured, strict CSP, `will-navigate`/`setWindowOpenHandler` locked down, Electron Fuses configured as claimed.
- **Preload whitelist** is exact-match against `contract.ts` — no dynamic channel construction possible, and cross-checking every `INVOKE_CHANNELS` entry against `registerAllHandlers()` found no orphaned or missing handlers, and no dead/unused ones.
- **Shell-escaping** is correctly and consistently double-escaped (arg-level then `sh -c`-level) across `RemoteShell`, `UnzipService`, `CompressService`, and `TerminalManager` — verified against real shells in tests for `$(...)`/backtick injection payloads.
- **Registry import** (`SessionImporter`/`regQuery`) uses `execFile` with an argv array (no shell), is depth-bounded against a hostile registry tree, and deliberately skips importing WinSCP passwords rather than risk a wrong plaintext.
- **`OperationRegistry.run()`'s** try/catch/finally guarantees every long-running operation reaches a terminal broadcast state — no code path leaves the UI hanging, even though the underlying SSH channel isn't always torn down (see Critical #7).
- No `v-html`/dynamic `innerHTML` anywhere in the file-browser or preview components — filenames/paths from a malicious server can't inject markup.
- `KeyGenerator`, `withRetry`, `fileSelection`/`fileSort` pure logic, `procParse.ts`, and the `CommandPalette` terminal-focus detection are all solid and well-tested where they exist.

## Recommended Fix Order

1. **`FilePane.vue` keyboard hijack (Critical #1/#2)** — real, easy-to-hit data loss during completely normal use (typing while files are selected). Fix first: scope `onKeydown` to ignore events whose `event.target` is a text-input element, and intersect `selectAll`/selection state with the filtered view.
2. **Path traversal (Critical #3)** and **host-key trust over-scoping (Critical #4)** — the two genuine security vulnerabilities reachable from a malicious/compromised server; block production use against untrusted servers until fixed.
3. **`EditSessionManager` leak (Critical #5)** and **`tailStreams` cross-session bug (Critical #6)** — both are real, reachable bugs with growing blast radius (unbounded disk/memory growth; misleading log output that could mask a real production issue).
4. **`RemoteShell` abort not destroying the stream (Critical #7)** and the tab-close session-orphan race (High-Priority #1) — connection/resource leaks that compound with extended use.
5. Everything else in High-Priority Suggestions, then Suggestions, roughly in listed order — none block correct operation under normal use, but several (zip argument injection, unguarded `decrypt()`, the terminal reattach races) are worth closing before the next release.
6. Backfill test coverage for the zero-coverage modules — prioritize `ipc/*.ipc.ts` (the raw-input boundary) and the two sanitizer functions already flagged as untested despite being security-relevant.
