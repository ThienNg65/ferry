# Ferry ‚Äî Project Map

## What this is

Ferry is a lightweight WinSCP alternative: an Electron + Vue 3 desktop app for SFTP file transfer with a dual-pane browser, a transfer queue (recursive ‚Äî whole folder trees, not just single files), and ‚Äî its main differentiators ‚Äî a genuinely live remote log tail (`tail -F` over SSH), a remote "Extract Here" archive-extraction action (zip and tar variants), browser-like site tabs (multiple concurrent SSH sessions, switchable instantly, deduped so re-connecting to an already-open site just switches to its tab), and an interactive SSH Terminal per site (auto-connects in the background as soon as the site connects, labeled `username@host`, opens directly at the site's configured remote start path if one is set), all wrapped in a minimal Apple-HIG-inspired UI built on `@nuxt/ui` v4. The file panes support click-to-sort columns, Explorer-style multi-select (Ctrl/Shift/Ctrl+A), a right-click context menu, F2 rename, and OS-level drag-and-drop (drop files/folders in from Explorer to upload; drag local rows out to Explorer via a native OS drag). SFTP/SSH only (no FTP/FTPS/SCP). Single-window, no `vue-router` (removed round-5 ‚Äî was an unused dependency) ‚Äî view (and tab) switching is plain reactive state.

## Tech stack

- Electron 30.5.1, electron-vite 2.x (build tool), electron-builder 24.x (packaging, NSIS)
- Vue 3.5 (Composition API, `<script setup>`), Pinia 2.1, TypeScript 5.4 (strict)
- `@nuxt/ui` 4.9.0 (installed from npm, NOT from the local source clone at `C:\toys\ui`) + Tailwind CSS v4 (`@tailwindcss/vite`)
- `ssh2` 1.16 (direct ‚Äî not `ssh2-sftp-client`, since raw exec channels are needed for tail/unzip alongside SFTP, and `client.shell()` for the interactive Terminal's PTY)
- `@xterm/xterm` 5.5 + `@xterm/addon-fit` 0.10 (renderer-side terminal emulator for the Terminal dock tab)
- `electron-store` (saved sites, plain JSON) + Electron's `safeStorage` (DPAPI-backed credential encryption ‚Äî no `keytar`)
- `@iconify/vue` + `@iconify-json/lucide` (full icon set pre-registered at startup ‚Äî see Gotchas)
- Node v22.14.0. npm installs on this machine need `--registry=https://registry.npmjs.org` explicitly.

## Entry points & key config

- `src/main/index.ts` ‚Äî main process bootstrap, `createWindow()`, `registerAllHandlers()`
- `src/renderer/src/main.ts` ‚Äî renderer bootstrap; **registers the full Lucide icon set here** (`addCollection`)
- `src/shared/contract.ts` ‚Äî change this FIRST when adding any new IPC channel; preload/main won't compile otherwise
- `electron.vite.config.ts` ‚Äî main/preload/renderer build config, path aliases (`@`, `@renderer`, `@shared`)
- `electron-builder.yml` ‚Äî packaging config. **Must be named exactly this** (see Gotchas)
- `tsconfig.node.json` / `tsconfig.web.json` ‚Äî split TS project references (main+preload+shared vs. renderer+shared)

## How the pieces talk to each other

- Renderer ‚Üí main: `window.api.invoke(channel, ...args)` ‚Üí preload validates channel against `INVOKE_CHANNELS`/`EVENT_CHANNELS` whitelist ‚Üí `ipcMain.handle` (registered via `handle()` in `ipc/envelope.ts`) ‚Üí always resolves `{ok:true,data}` or `{ok:false,code,message}`. Renderer's `api.ts` `invoke()` unwraps this and throws on `ok:false`.
- Main ‚Üí renderer: `webContents.send(EVENT_CHANNELS.x, payload)` broadcast to all windows; renderer subscribes via `window.api.on(channel, cb)` (`onEvent()` wrapper).
- All SSH operations for a connection go through one `RemoteShell` instance (wraps a single `ssh2.Client`), owned by `SessionManager`, keyed by a renderer-issued `sessionId` (UUID) ‚Äî NOT tied to a saved site id, so the same site can have multiple concurrent sessions (e.g. browsing + a tail), and the renderer's site-tabs feature keeps one `sessionId` alive per open tab.
- `TailManager`, `TerminalManager`, and `TransferQueue` are separate singletons that look up the `RemoteShell` for a given `sessionId` via `SessionManager.getInstance().shell(sessionId)` ‚Äî they don't own connections themselves. `SessionManager.close(sessionId)` calls `stopAllForSession`/`closeAllForSession` on both `TailManager` and `TerminalManager` before ending the client ‚Äî add the same hook here for any future per-session subsystem.

## Conventions

- **IPC contract-first**: every new feature starts by adding channels/types to `src/shared/contract.ts`, then an `ipc/*.ipc.ts` handler file with `registerXHandlers()`, then wire it into `registerAllHandlers()` in `main/index.ts`.
- **fs/ services are plain exported functions, not classes** (stateless) ‚Äî unlike `SessionManager`/`SiteStore`/`TailManager`/`TransferQueue`/`TerminalManager`, which ARE singletons (`getInstance()`) because they hold state.
- **Secrets never cross the IPC boundary decrypted.** `SiteStore` only returns `hasPassword`/`hasPassphrase` booleans to the renderer; decryption happens main-process-side, in-memory, at connect time only.
- **No silent auto-reconnect for browsing sessions** (stale listings/in-flight transfers would be unsafe) ‚Äî only `TailManager` auto-reconnects, since re-tailing is safe.
- **Icons**: use `i-lucide-*` names anywhere (`UIcon`, `icon` props, even computed/dynamic bindings) ‚Äî all Lucide icons are pre-registered offline, no need to worry about static-analysis detection.
- **Drag-and-drop has two distinct mechanisms, distinguished at the drop target by `event.dataTransfer.types.includes('Files')`.** Remote-row-to-local-pane (download) still uses a module-scoped Vue `ref` (`useDragAndDrop.ts`) carrying `{side, entry}` as in-app-only shared state (`dataTransfer` only carries `text/plain`, since there's no local file yet to hand the OS). Local rows instead call `system:startDrag` (new IPC, see below) on `dragstart`, which hands the real file path to `WebContents.startDrag` ‚Äî this produces a native OS drag that (a) can be dropped onto Explorer/Finder or another app, and (b) is *also* what the remote pane's drop handler sees when the same drag lands back inside the app, alongside genuine OS-Explorer-file drops ‚Äî both are treated identically as an upload (`FilePane.vue`'s `onOsDrop`). Never assume `getDragPayload()` is populated for a local-row drag anymore ‚Äî check `isOsFileDrag(event)` first.
- **`ipc/envelope.ts` has a second registration helper, `handleWithEvent()`, alongside `handle()`.** Use it (instead of `handle()`) only when a handler needs the raw `IpcMainInvokeEvent` itself ‚Äî today that's just `system:startDrag` (`event.sender.startDrag(...)`). Everything else should keep using plain `handle()`.
- **A component not yet emitted by `unplugin-vue-components`' scanner (i.e. missing from `src/renderer/components.d.ts`) is imported explicitly** via the package's public `./components/*` export path (e.g. `import ContextMenu, { type ContextMenuItem } from '@nuxt/ui/components/ContextMenu.vue'`) rather than relying on global `<UContextMenu>` registration + ambient types, which only exist for components some template has already used at the time `.d.ts` was generated. `vue-tsc --build` (used by `npm run typecheck`, which does NOT run Vite) never regenerates that file, so a genuinely new global-only usage would fail to type-check until the dev/build Vite pass runs once. `FileRow.vue`'s context menu is the first user of this pattern ‚Äî follow it for the next not-yet-used Nuxt UI component instead of guessing at global registration.
- **Dialogs use `UModal`** (`v-model:open` or `:open` + `@update:open`, `#body`/`#footer="{ close }"` slots): `SiteFormDialog.vue`, `FilePreviewDialog.vue`, and the inline delete-confirm modal in `SessionManagerView.vue`. `UModal` also has an `#actions` slot rendered next to its built-in close button ‚Äî used by `FilePreviewDialog.vue` for the tail-shortcut button instead of a fully custom `#header`. `USlideover` is reserved for contextual drill-down panels, not bounded CRUD forms ‚Äî not used anywhere yet.
- **Toasts**: call `useNotify()` (`src/composables/useNotify.ts`), never `useToast()` directly ‚Äî keeps styling (`color`, `icon`) consistent across call sites. Toast only on outcomes the user didn't just get obvious visual feedback for (transfer done/error, extract done/error, connect success/error, unexpected disconnect) ‚Äî deliberately NOT on every fs op, site CRUD, or user-initiated disconnect; those already have an immediate visible UI change and toasting them would be noise.
- **Window chrome is fully custom** ‚Äî `titleBarStyle: 'hidden'` alone (no `titleBarOverlay`, no `frame: false`), real Vue `<button>`s in `TitleBar.vue` wired to `window:minimize`/`window:maximizeToggle`/`window:close`/`window:isMaximized` IPC + a `window:state-change` broadcast event. Plain `<button>`, not `UButton`, for these three ‚Äî `UButton`'s rounded/padded default styling doesn't match the flush, 44px-wide, square-hover-target Windows caption-button convention.
- **`main/index.ts` keeps a module-level `mainWindow` reference** (not just a local inside `createWindow()`) so IPC handlers (`window.ipc.ts`) and window event listeners (`maximize`/`unmaximize` ‚Üí broadcast) can both reach the single app window.
- **Site tabs are a renderer-only concept.** `sessions.store.ts`'s state is `SessionTab[]` (`tabId`, `sessionId | null` while in "picker" state, `siteId | null` ‚Äî set only for saved-site connects, used to dedup ‚Äî `label`, `hostLabel` ‚Äî `username@host`, shown by `TerminalView.vue` ‚Äî `status`, `statusMessage`, `connecting`) + `activeTabId`; there is always ‚â•1 tab (closing the last one opens a fresh picker tab rather than leaving zero). `activeTab`/`activeSessionId`/`status`/`statusMessage`/`connecting` are **getters** that resolve from the active tab ‚Äî this keeps `App.vue`/`FilePane.vue`/`SessionManagerView.vue` reading the exact same flat names they always did, with zero template changes. If you add a new such getter, never let anything assign to it directly (getters can't be assigned) ‚Äî route mutations through an action that finds the specific tab object and mutates it (`tab.status = ...`), never `this.status`.
- **`connectToSite(site)` dedups by `siteId`.** Before opening a new connection it checks `tabs` for an existing tab with the same `siteId` and `status === 'connected' || connecting`; if found it just `setActiveTab`s to it (and cleans up the calling tab if that tab was an unused empty picker) instead of opening a duplicate session. Quick-connect (`connect()`, ad-hoc, `siteId: null`) is deliberately NOT deduped ‚Äî there's no stable identity to dedup an ad-hoc host on. A previously **errored** tab for a site is also deliberately NOT matched (only `connected`/`connecting` are) ‚Äî clicking Connect again opens a fresh tab rather than reusing/hiding the dead one, so the user can still see what failed.
- **The interactive Terminal now auto-opens in the background as soon as a site connects** (`sessions.store.ts`'s `openSession()` fires `terminalStreams.ensureTerminal(sessionId)` without awaiting, right after a successful connect) rather than waiting for the user to click the Terminal dock tab. Because of this, `terminalStreams.store.ts`'s `ensureTerminal()` must be safe against being called twice for the same session before the first call's `terminal:open` round-trip resolves (the auto-open racing a user's immediate click) ‚Äî it tracks in-flight opens in a `pending: Map<string, Promise<string>>` alongside the resolved `instances` map, so a second overlapping call awaits the same promise instead of opening a second PTY. Keep this pattern if you add another eager/automatic call site for `ensureTerminal`.
- **`remoteFs.store.ts` is keyed by `sessionId`** (`bySession: Record<string, PerSessionFs>`) so each open site tab keeps its own independent `currentPath`/`entries`/`selected`/etc., but exposes the identical flat `currentPath`/`entries`/`loading`/`error`/`selected` getter surface as before (resolved via a `current` getter reading the active session's bucket, falling back to a frozen `EMPTY_DEFAULT` when no bucket exists yet) ‚Äî `FilePane.vue` needed no template changes for this either. Lazy-create a session's bucket only inside actions (e.g. `ensureBucket()` at the top of `load()`), never inside a getter (Pinia/Vue getters must stay pure ‚Äî no side effects). There's a real (intentional, safe) circular import between `sessions.store.ts` and `remoteFs.store.ts`/`terminalStreams.store.ts` (each calls the other's `useXStore()` inside action bodies, never at module-eval time) ‚Äî this is a standard, working ESM circular-import pattern for cross-referencing Pinia stores; don't "fix" it by breaking the reference apart.
- **Live xterm.js `Terminal` instances live in a plain module-scoped `Map` inside `terminalStreams.store.ts`, deliberately OUTSIDE Pinia's reactive state**, keyed by `sessionId` (one terminal per session). A single `terminal:data` IPC subscription (set up once, lazily) routes each chunk to the right cached instance ‚Äî components never subscribe individually. `TerminalView.vue` only ever attaches/shows the DOM (`term.open(el)` exactly once ever per instance, guarded by a local `Set`); it never constructs or disposes a `Terminal` itself. This is the pattern to follow for any future feature that needs a stateful, non-serializable client object (a socket, a canvas context, a third-party widget) to survive across tab/dock-panel switching without being destroyed by Vue's `v-if`.
- **A dock tab that must survive `BottomDock.vue`'s own tab-switching (and its collapse toggle) uses `v-show`, not `v-if`/`v-else-if`.** Transfers/Log Tail are stateless-enough to freely remount (the existing `v-if`/`v-else-if` chain), but `TerminalView` sits in its own always-present sibling `<div v-show="!collapsed && tab === 'terminal'">` (mounted once, guarded by a `terminalEverShown` flag) specifically so switching dock tabs or collapsing the dock never tears down its live xterm instances.
- **The Local-pane hide toggle lives in `SiteTabBar.vue`, pinned to the right end** (a `<div class="flex-1">` spacer separates it from the tab chips + "+" button), not in `FilePane.vue`'s own header ‚Äî but the collapse/expand logic itself (`collapsedRail`/`showBody` computed pair, the fixed `h-8` header row on both sides, the watcher that closes any open preview dialog on collapse) still lives entirely in `FilePane.vue`/`ui.store.ts`; only the trigger button's location moved. Hiding the Local pane still collapses it to a slim rail, it never fully unmounts.
- **Icon-only buttons get wrapped in `<UTooltip text="...">`, everywhere** (reusing existing `aria-label` text where present). `UTooltip`'s default slot renders its trigger via `asChild` (reka-ui) ‚Äî no extra wrapper DOM node is added, so it's safe to wrap hover-reveal buttons (`class="opacity-0 group-hover:opacity-100"`) without breaking that pattern. Buttons with a visible text label (e.g. `BottomDock.vue`'s Transfers/Log Tail/Terminal tab buttons) don't need one.
- **`FileRow.vue`'s per-row action icons (tail/extract/transfer/delete) each sit in their own fixed-width `w-7` slot**, not bare conditionally-rendered buttons ‚Äî because which icons apply varies per row (a directory shows none of tail/extract/transfer, only delete; a non-zip file skips extract), a bare `v-if` would let the row's flex layout collapse and shift later icons (most visibly, `delete`) leftward depending on how many earlier icons applied. Wrapping each action in `<div class="flex w-7 shrink-0 items-center justify-center">` (with the actual button/tooltip `v-if`'d inside) reserves the slot's width even when empty, so every row's icons land in identically-positioned columns. The same fixed-width-column technique is used for the remote-only permissions cell (`w-36`, with an equal-width empty placeholder div when a row's `side !== 'remote'`) ‚Äî follow this pattern for any future per-row column that isn't always present/isn't fixed-length text.
- **Remote-only file permissions**: `FileEntry.permissions` (a 4-digit octal string, e.g. `"0755"`) is populated only by `RemoteFsService`/`RemoteShell.ts` ‚Äî `LocalFsService.ts` never sets it, since local (Windows) `fs.stat().mode` isn't meaningful POSIX data. `renderer/src/utils/permissions.ts` formats it two ways: `toTechnical()` (classic `rwxr-xr-x`, all three owner/group/other triplets) and `toFriendlyLabel()` (a single compact phrase derived from the OWNER triplet only, e.g. "Read & Write" ‚Äî explicitly scoped to the owner, not the connected user's own effective access, since those can differ on a shared server; the full technical string is always one hover away via the cell's tooltip). `ui.store.ts`'s `permissionsDisplay` (persisted, default `'friendly'`) picks which one `FileRow.vue` renders; toggled via `FileToolbar.vue`'s permissions-toggle button (remote pane only, gated by its own `side` prop).
- **Custom scrollbar styling reuses `@nuxt/ui`'s own runtime CSS vars** (`--ui-border-accented`, `--ui-color-neutral-400`/`500`) in `main.css`'s global `::-webkit-scrollbar*` rules, rather than hardcoding new colors ‚Äî keeps the scrollbar in sync with the app's theme automatically, the same way the file's pre-existing `--ui-primary` override already does.
- **Deferred-but-still-full icon-set loading, and lazy-loaded post-connect components, keep cold start off the JS-parse critical path.** `main.ts` no longer statically imports `@iconify-json/lucide/icons.json` (a static import is hoisted and parsed before any other top-level code runs, blocking first paint, regardless of where the `addCollection()` *call* sits in the file) ‚Äî it's a dynamic `import()` scheduled via `requestIdleCallback`/`setTimeout` fallback, run *after* `mount()`. It still registers the WHOLE set (not a hand-picked subset) because some `i-lucide-*` names are bound dynamically (computed props, ternaries) and Vite's static icon-bundling can't detect those ‚Äî icons requested before the deferred load resolves just pop in a frame later, which Iconify's runtime handles natively. Same pattern for `@xterm/xterm`/`@xterm/addon-fit`: type-only imports at the top of `terminalStreams.store.ts`, the real classes dynamically imported inside `ensureTerminal()` right before construction. And `App.vue`'s `FilePane`/`BottomDock` (only ever rendered post-connect) are `defineAsyncComponent(() => import(...))` rather than static imports ‚Äî `TitleBar`/`SiteTabBar`/`SessionManagerView` stay static since they render at first paint. If you add another library/component that's expensive to parse but not needed until some later user action, follow this same "type-only import + dynamic import at the point of use" shape rather than a static import.
- **Shell-command construction shares one escaping helper.** `src/main/ssh/shellEscape.ts` (single-quote POSIX escaping, embedded `'` ‚Üí `'\''`) is imported by both `UnzipService.ts` (building `unzip`/`tar` commands) and `TerminalManager.ts` (injecting a `cd` on terminal open) ‚Äî don't reintroduce a local private copy in a new call site; import this one.
- **Archive-extension recognition lives in `src/shared/archive.ts`, not hardcoded per call site.** `archiveKind()`/`isArchive()` are pure functions (no Electron/Node deps) imported by both the renderer (`fileTypes.ts`, re-exported for `FileRow.vue`'s "Extract here" gating) and the main process (`UnzipService.ts`, choosing `unzip` vs. `tar -xf`/`-xzf`/`-xjf` per kind) ‚Äî this is the shared-module pattern to follow for any future renderer/main-agreed classification logic (mirrors how `contract.ts` itself is shared, but scoped to domain logic rather than IPC shapes). `.7z`/`.rar` are deliberately unrecognized (not reliably available on remote hosts) ‚Äî don't add them without also designing new missing-tool error handling.
- **The Terminal lands in the site's configured remote start path, not the SSH default.** SSH's shell-channel protocol has no cwd concept at all (confirmed against `@types/ssh2`) ‚Äî `TerminalManager.open()` writes a `cd <shellEscape(path)>\n` into the freshly-opened stream (after listeners are attached, so the shell's response is forwarded like any typed command) when `SessionManager.cwd(sessionId)` resolves to something other than `.`. That value is already seeded from `Site.remoteInitialPath` at connect time for the file browser's benefit ‚Äî this just added a second consumer, no new plumbing. If you add another feature that needs a session's configured start path, read `SessionManager.cwd(sessionId)`, don't add new IPC plumbing for it.
- **One shared `.fade-*` CSS transition (`main.css`) covers every simple show/hide in the app** ‚Äî the title-bar busy dot and `App.vue`'s picker/connected swap both use `<Transition name="fade">` with these classes rather than each defining their own. Reuse it for future similar cases instead of writing a new one-off transition.
- **Do not manufacture a transition by re-keying/re-mounting `FilePane`/`BottomDock`/`TerminalView` per active site tab.** Switching between two already-connected tabs deliberately does NOT swap those components ‚Äî same instances, different session data via computed getters (see the `remoteFs.store.ts`/`terminalStreams.store.ts` entries above) ‚Äî forcing a remount there to animate the switch would tear down live xterm scrollback and cached fs listings that this architecture specifically preserves. The only top-level state boundary that gets a fade is the picker‚Üîconnected swap in `App.vue`.
- **CSS transitions on a collapse/expand toggle need explicit sizes on BOTH states ‚Äî a `flex-1`/no-explicit-value side will not animate.** `FilePane.vue`'s Local-pane rail collapse animates `width` between `w-10` (collapsed) and `min-w-0 w-1/2` (expanded, LOCAL side only ‚Äî the ternary is conditioned on `side === 'local'`, since `FilePane.vue` is instantiated twice from one template and the Remote pane must keep its own unrelated `flex-1` untouched in both of Local's states, or it won't grow to fill the space Local's rail frees up). `min-w-0 flex-1` (flex-grow-derived, no explicit `width`, computed style stays `width: auto`) was tried first and produced NO visible animation at all ‚Äî CSS cannot interpolate to/from an `auto`-resolved width, only between two concrete `<length-percentage>` values (`BottomDock.vue`'s dock collapse works for the same reason: `h-9`/`h-56` are both explicit). If a future pane/panel collapse needs the same treatment, give both states an explicit length ‚Äî never leave one side as bare `flex-1`/`auto`.
- **A generic in-flight IPC-call counter (`ipcActivity.store.ts`), not hand-picked per-action flags, drives the title-bar busy indicator.** `api.ts`'s `invoke()` increments/decrements this store's `count` around every call (skipping a small `QUIET_CHANNELS` exclusion set: `terminal:write`/`terminal:resize` ‚Äî fire per keystroke/resize ‚Äî and the four window-chrome channels, semantically unrelated to "SSH/SFTP work"). `useGlobalActivity.ts`'s `isBusy` is primarily this counter (`ipcActivity.isBusy`), so any new IPC channel is automatically covered without remembering to wire it in ‚Äî the composable's `connecting`/transfer-count/fs-`loading` checks exist only to pick a nicer LABEL when they're the active reason, falling back to a generic "Working‚Ä¶" otherwise. If you add a new high-frequency channel (fires on a timer or per-keystroke/per-frame), add it to `QUIET_CHANNELS`, don't leave it tracked.
- **A composable that returns multiple refs (e.g. `useGlobalActivity()`'s `{ isBusy, label }`) must be destructured at the call site, not accessed via the returned object's dot-notation, if used directly in a template.** `const activity = useGlobalActivity(); ...{{ activity.isBusy }}...` does NOT auto-unwrap ‚Äî `activity` is a plain object, and Vue's template compiler only auto-unwraps refs that are top-level `<script setup>` bindings (or accessed through a `reactive()`-wrapped object), not nested properties reached via dot-access on a plain object. Always write `const { isBusy, label } = useGlobalActivity()` and reference `isBusy`/`label` directly in the template. Getting this wrong doesn't error at build time ‚Äî it silently passes the raw `Ref` object as a prop/condition (a `v-if` on a raw ref object is always truthy, and a `:prop="someRef"` binding throws a runtime "invalid prop type" warning), so watch for that failure shape if a composable's boolean/value never seems to update in a template.
- **Recursive directory transfer is one `TransferQueue` job, not N.** `enqueueTree()` pushes a `TransferJob` with `isTree: true` into the SAME `pending`/`active` queue as single-file jobs (so the existing `MAX_CONCURRENT`/cancel machinery needs no changes), but `runTree()` then walks the whole source tree (`LocalFsService.listRecursive`/`RemoteShell.readdirRecursive`), recreates every directory at the destination (`mkdir -p`-equivalent, tolerant of already-existing dirs), and transfers its files **sequentially, one at a time** ‚Äî not fanned out to the pool. `runFile()` (the byte-streaming core, shared by both a solo file job and each file inside a tree job) takes an `onProgress` callback rather than broadcasting directly, so `runTree()` can aggregate all its files' bytes into ONE throttled `progress` event under the job's single `transferId` ‚Äî the renderer never learns a tree transfer contains more than one file. If you need per-tree concurrency (multiple files of the same tree in flight at once) that's a deliberate follow-up, not an oversight ‚Äî sequential was chosen to keep this job type simple and to not blow past `MAX_CONCURRENT` real SFTP streams when several tree jobs happen to be active together.
- **File-list sort and selection logic are pure functions, not inlined into the stores.** `renderer/src/utils/fileSort.ts`'s `compareEntries(column, direction)` and `fileSelection.ts`'s `selectOnly`/`toggleSelect`/`selectRange`/`selectAll` are imported identically by both `localFs.store.ts` (flat state) and `remoteFs.store.ts` (per-session `PerSessionFs` bucket) ‚Äî each store just assigns the pure function's result onto its own state shape. `selectRange` needs a `selectAnchor` (the last plain-click or ctrl-click target) alongside `selected`; both stores carry it, reset to `null` on every fresh `load()`. Keep new sort/selection behavior in these shared modules, not duplicated per store, if a third fs-like store is ever added.

## Gotchas that will waste your first hour

1. **`electron-builder.yml` must be named EXACTLY that** (or `electron-builder.ts`/`.json`/etc.) ‚Äî NOT `electron-builder.config.ts`. electron-builder's config loader (`read-config-file`) only searches for `electron-builder.<ext>`; `*.config.ts` is silently ignored with no error, and you get default settings (`oneClick: true`, wrong product name, no asarUnpack). We hit this for real ‚Äî see `.claude/plan/ferry-implementation-plan.md` history. Also: electron-builder's `.ts`-config loader (`config-file-ts`) has a Windows bug where absolute paths with a drive letter break its cache-directory naming ‚Äî prefer `.yml` over `.ts` for this config on Windows regardless.
2. **Icons will silently 404/CSP-block unless the full Lucide set is registered ‚Äî AND the network-fallback path is disabled, not just deferred.** `@nuxt/ui`'s icon component falls back to fetching unrecognized icon names from Iconify's public API at runtime ‚Äî blocked by our CSP (`default-src 'self'`), and Vite's static icon-bundling can't detect dynamically-bound icon names (computed props, ternaries) anyway. `main.ts` registers the full Lucide set via a dynamic `import()` deferred to `requestIdleCallback` (so cold-start isn't blocked on parsing 554KB of JSON) ‚Äî but that alone isn't sufficient, since icons rendered in the first frame (before the deferred load resolves) would still hit the network-fallback path and get CSP-blocked. `main.ts` also calls `setCustomIconLoader(loader, 'lucide')` (from `@iconify/vue`) with a loader that `await`s the SAME memoized load-then-`addCollection()` promise used by the idle-scheduled call, returning `null` once it resolves ‚Äî this is NOT a no-op/never-resolving loader (a loader that never settles would permanently strand any icon that hit it before the real load finished, since Iconify only re-notifies already-mounted `Icon` instances when a load attempt actually *settles*, not merely when `addCollection()` is later called on its own). If you add a different icon collection, register both halves (the deferred `addCollection` load AND a `setCustomIconLoader` gating on that same promise) the same way ‚Äî don't just add a bare `addCollection()` call.
3. **`titleBarStyle: 'hidden'` alone removes ALL window chrome on Windows** ‚Äî no close/minimize/maximize button, only Alt+F4 ‚Äî and that's now intentional: `TitleBar.vue` draws its own buttons wired to `window:*` IPC (see Conventions). Do NOT re-add `titleBarOverlay` "to be safe" ‚Äî it draws real OS caption buttons over the custom bar with no way to hide just the buttons while keeping the overlay, which is exactly what we removed it to avoid.
4. **`tsconfig.web.json`'s `include` must cover `src/renderer/auto-imports.d.ts`, not just `src/renderer/src/**/*`.** `unplugin-auto-import` (via `@nuxt/ui/vite`) writes global ambient declarations (`useToast`, `useOverlay`, etc.) to that file, one directory above the renderer source root. Component tags (`UButton`, `UModal`, ...) still resolve fine in `<template>` even without it, because Vue's template checker (`@vue/language-core`) discovers `components.d.ts`-style files through a separate mechanism ‚Äî but a bare composable call in `<script setup>` (e.g. `useToast()`) goes through the real TS program and needs the ambient file in `include` to type-check. Hit this for real adding `useNotify.ts` ‚Äî fixed by adding the path explicitly.
5. **OS drag-and-drop (`system:startDrag`, reading `File.path` off dropped items) depends on an Electron API surface that changes at v32.** This project is pinned to Electron 30.5.1, where the legacy `File.path` extension (used in `FilePane.vue`'s `onOsDrop` and implicitly by `WebContents.startDrag`'s renderer-side trigger) still works under `sandbox: true`. Electron 32 introduces `webUtils.getPathForFile()` as the sandboxed-safe replacement and begins deprecating `File.path`; upgrading Electron past 32 will need `onOsDrop` migrated to call `webUtils.getPathForFile(file)` via a small preload-exposed helper instead of reading `.path` directly off the DOM `File` object. Don't upgrade Electron without checking this first ‚Äî OS drag-and-drop would silently stop working (no path, no upload) rather than erroring loudly.
6. **App icon is now set** ‚Äî `resources/1024x1024.png` is electron-builder's `win.icon` source (auto-generates every Windows resolution from one square PNG), `resources/favicon.ico` is the `BrowserWindow`'s dev-mode-only icon (`main/index.ts`, gated on `IS_DEV` since packaged builds inherit the exe's embedded icon instead), and `src/renderer/public/favicon.ico` (referenced via a relative `<link rel="icon">` in `index.html`) covers the (mostly invisible, given the custom titlebar) renderer-side favicon.
7. `@toys\ui` (the local `@nuxt/ui` source clone) is NOT a dependency of this project and has no build output ‚Äî it's reference material only (source browsing + the `nuxt-ui` skill). Never point tooling at it.
8. **File preview/download/tail affordances only apply to the remote pane, never local.** `FilePreviewDialog.vue`'s download and tail buttons are gated on `side === 'remote'` ‚Äî downloading or tailing a file that's already on the local machine doesn't make sense, and `TailManager`/`tail:start` only ever operates over an SSH session. If you add a similar affordance elsewhere, gate it the same way (mirrors `FilePane.vue`'s pre-existing `showTail = side === 'remote' && connected` computed).
9. **Never call `xterm.js`'s `term.open()` more than once per `Terminal` instance.** Reparenting a live terminal onto a second element isn't a documented-safe operation, so `TerminalView.vue` guards it with a `Set` of already-attached session ids and relies on the container staying permanently in the DOM (`v-show`, see Conventions) instead. If a future refactor needs the container to actually unmount/remount, dispose the old `Terminal` and create a fresh one rather than re-opening it.
10. **Terminal output crosses IPC as raw `Uint8Array`, not a decoded string.** `TerminalManager` forwards each `stream.on('data', chunk: Buffer)` as `new Uint8Array(chunk)` rather than `chunk.toString()`, because a naive per-chunk UTF-8 decode can split a multi-byte character across two chunks and corrupt it; `xterm.js`'s `Terminal.write()` accepts `Uint8Array` directly so no decode step is needed in the renderer either. Keystrokes going the other way (`terminal:write`) are a plain string ‚Äî that direction has no chunk-boundary problem.
