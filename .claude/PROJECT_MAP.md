# Ferry — Project Map

## What this is

Ferry is a lightweight WinSCP alternative: an Electron + Vue 3 desktop app for SFTP file transfer with a dual-pane browser, a transfer queue (recursive — whole folder trees, not just single files, with per-row retry on failure, and an app-wide bandwidth cap — see Settings), and — its main differentiators — a genuinely live remote log tail (`tail -F` over SSH), a remote "Extract Here" archive-extraction action (zip and tar variants) plus its inverse "Compress to .zip" (local via `archiver`, remote via SSH-exec `zip -r`, both single-entry), browser-like site tabs (multiple concurrent SSH sessions, switchable instantly, deduped so re-connecting to an already-open site just switches to its tab, and restored — as picker tabs, not auto-connected — across app restarts), a Ctrl/Cmd+K command palette, a light/dark theme toggle, and an interactive SSH Terminal per site (auto-connects in the background as soon as the site connects, labeled `username@host`, opens directly at the site's configured remote start path if one is set), all wrapped in a minimal Apple-HIG-inspired UI built on `@nuxt/ui` v4. The file panes support click-to-sort columns, Explorer-style multi-select (Ctrl/Shift/Ctrl+A), a right-click context menu, F2 rename, editable permissions (chmod), per-file-type icons, and OS-level drag-and-drop (drop files/folders in from Explorer to upload; drag local rows out to Explorer via a native OS drag). Saved sites can be grouped (free-text tag), searched, duplicated, and imported in bulk from WinSCP/PuTTY's saved sessions (Windows registry scan; passwords are never imported, by design). Auth supports password, private key, SSH agent, and connecting through a jump/bastion host; every connection is host-key-verified (trust-on-first-use, warns on change) and real keyboard-interactive/2FA challenges are forwarded to the user instead of being guessed at. Packaged builds check GitHub Releases for updates on launch (electron-updater, currently inert scaffolding — see Gotcha below) and notify via toast when one's ready to install. SFTP/SSH only (no FTP/FTPS/SCP). Single-window, no `vue-router` (removed round-5 — was an unused dependency) — view (and tab) switching is plain reactive state.

## Entry points & key config

- `src/main/index.ts` — main process bootstrap, `createWindow()`, `registerAllHandlers()`
- `src/renderer/src/main.ts` — renderer bootstrap; **registers the full Lucide icon set here** (`addCollection`)
- `src/shared/contract.ts` — change this FIRST when adding any new IPC channel; preload/main won't compile otherwise
- `electron.vite.config.ts` — main/preload/renderer build config, path aliases (`@`, `@renderer`, `@shared`)
- `electron-builder.yml` — packaging config. **Must be named exactly this** (see Gotchas)
- `tsconfig.node.json` / `tsconfig.web.json` — split TS project references (main+preload+shared vs. renderer+shared)

## How the pieces talk to each other

- Renderer → main: `window.api.invoke(channel, ...args)` → preload validates channel against `INVOKE_CHANNELS`/`EVENT_CHANNELS` whitelist → `ipcMain.handle` (registered via `handle()` in `ipc/envelope.ts`) → always resolves `{ok:true,data}` or `{ok:false,code,message}`. Renderer's `api.ts` `invoke()` unwraps this and throws on `ok:false`.
- Main → renderer: `webContents.send(EVENT_CHANNELS.x, payload)` broadcast to all windows; renderer subscribes via `window.api.on(channel, cb)` (`onEvent()` wrapper).
- All SSH operations for a connection go through one `RemoteShell` instance (wraps a single `ssh2.Client`), owned by `SessionManager`, keyed by a renderer-issued `sessionId` (UUID) — NOT tied to a saved site id, so the same site can have multiple concurrent sessions (e.g. browsing + a tail), and the renderer's site-tabs feature keeps one `sessionId` alive per open tab.
- `TailManager`, `TerminalManager`, `TransferQueue`, `OperationRegistry`, and `MonitorManager` are separate singletons that look up the `RemoteShell` for a given `sessionId` via `SessionManager.getInstance().shell(sessionId)` — they don't own connections themselves. `SessionManager.close(sessionId)` calls `stopAllForSession`/`closeAllForSession`/`cancelAllForSession` on `TailManager`, `TerminalManager`, `OperationRegistry`, and `MonitorManager` before ending the client — add the same hook here for any future per-session subsystem.

Each integration file does an async top-level `probeServer()` before its `describe.skipIf(!serverAvailable)(...)` — if the container isn't running, the suite self-skips with a console warning instead of failing. Any file that touches `SessionManager`/`TransferQueue` (and transitively `TailManager`/`TerminalManager`, all real, unmocked) needs `vi.mock('electron', () => ({ BrowserWindow: { getAllWindows: () => [] } }))` at the top, since these singletons broadcast IPC events via `BrowserWindow.getAllWindows()` and vitest runs outside a real Electron process — follow this pattern (mock only `BrowserWindow`, nothing else) for any future integration test that touches a main-process singleton. **`fileParallelism: false` is load-bearing, not a style choice**: `SessionManager.integration.test.ts`, `SessionManager.jumphost.integration.test.ts`, and `TransferQueue.integration.test.ts` all legitimately touch the SAME real disk-backed `known_hosts.json` (via `KnownHostsStore`) for the SAME test-server host:port — running test *files* concurrently (vitest's default) raced on that shared state and produced a real, reproducible flaky failure before this was added; if you ever see `KnownHostsStore`-related test flakiness again, check this setting hasn't been reverted before debugging anything else.

## Conventions

- **IPC contract-first**: every new feature starts by adding channels/types to `src/shared/contract.ts`, then an `ipc/*.ipc.ts` handler file with `registerXHandlers()`, then wire it into `registerAllHandlers()` in `main/index.ts`.
- **fs/ services are plain exported functions, not classes** (stateless) — unlike `SessionManager`/`SiteStore`/`TailManager`/`TransferQueue`/`TerminalManager`, which ARE singletons (`getInstance()`) because they hold state.
- **Secrets never cross the IPC boundary decrypted.** `SiteStore` only returns `hasPassword`/`hasPassphrase` booleans to the renderer; decryption happens main-process-side, in-memory, at connect time only.
- **No silent auto-reconnect for browsing sessions** (stale listings/in-flight transfers would be unsafe) — only `TailManager` auto-reconnects, since re-tailing is safe.
- **Icons**: use `i-lucide-*` names anywhere (`UIcon`, `icon` props, even computed/dynamic bindings) — all Lucide icons are pre-registered offline, no need to worry about static-analysis detection.
- **Drag-and-drop has two distinct mechanisms, distinguished at the drop target by `event.dataTransfer.types.includes('Files')`.** Remote-row-to-local-pane (download) still uses a module-scoped Vue `ref` (`useDragAndDrop.ts`) carrying `{side, entry}` as in-app-only shared state (`dataTransfer` only carries `text/plain`, since there's no local file yet to hand the OS). Local rows instead call `system:startDrag` (new IPC, see below) on `dragstart`, which hands the real file path to `WebContents.startDrag` — this produces a native OS drag that (a) can be dropped onto Explorer/Finder or another app, and (b) is *also* what the remote pane's drop handler sees when the same drag lands back inside the app, alongside genuine OS-Explorer-file drops — both are treated identically as an upload (`FilePane.vue`'s `onOsDrop`). Never assume `getDragPayload()` is populated for a local-row drag anymore — check `isOsFileDrag(event)` first.
- **`ipc/envelope.ts` has a second registration helper, `handleWithEvent()`, alongside `handle()`.** Use it (instead of `handle()`) only when a handler needs the raw `IpcMainInvokeEvent` itself — today that's just `system:startDrag` (`event.sender.startDrag(...)`). Everything else should keep using plain `handle()`.
- **A component not yet emitted by `unplugin-vue-components`' scanner (i.e. missing from `src/renderer/components.d.ts`) is imported explicitly** via the package's public `./components/*` export path (e.g. `import ContextMenu, { type ContextMenuItem } from '@nuxt/ui/components/ContextMenu.vue'`) rather than relying on global `<UContextMenu>` registration + ambient types, which only exist for components some template has already used at the time `.d.ts` was generated. `vue-tsc --build` (used by `npm run typecheck`, which does NOT run Vite) never regenerates that file, so a genuinely new global-only usage would fail to type-check until the dev/build Vite pass runs once. `FileRow.vue`'s context menu is the first user of this pattern — follow it for the next not-yet-used Nuxt UI component instead of guessing at global registration.
- **Dialogs use `UModal`** (`v-model:open` or `:open` + `@update:open`, `#body`/`#footer="{ close }"` slots): `SiteFormDialog.vue`, `FilePreviewDialog.vue`, and the inline delete-confirm modal in `SessionManagerView.vue`. `UModal` also has an `#actions` slot rendered next to its built-in close button — used by `FilePreviewDialog.vue` for the tail-shortcut button instead of a fully custom `#header`. `USlideover` is reserved for contextual drill-down panels, not bounded CRUD forms — not used anywhere yet.
- **Toasts**: call `useNotify()` (`src/composables/useNotify.ts`), never `useToast()` directly — keeps styling (`color`, `icon`) consistent across call sites. Toast only on outcomes the user didn't just get obvious visual feedback for (transfer done/error, extract done/error, connect success/error, unexpected disconnect) — deliberately NOT on every fs op, site CRUD, or user-initiated disconnect; those already have an immediate visible UI change and toasting them would be noise.
- **Window chrome is fully custom** — `titleBarStyle: 'hidden'` alone (no `titleBarOverlay`, no `frame: false`), real Vue `<button>`s in `TitleBar.vue` wired to `window:minimize`/`window:maximizeToggle`/`window:close`/`window:isMaximized` IPC + a `window:state-change` broadcast event. Plain `<button>`, not `UButton`, for these three — `UButton`'s rounded/padded default styling doesn't match the flush, 44px-wide, square-hover-target Windows caption-button convention.
- **`main/index.ts` keeps a module-level `mainWindow` reference** (not just a local inside `createWindow()`) so IPC handlers (`window.ipc.ts`) and window event listeners (`maximize`/`unmaximize` → broadcast) can both reach the single app window.

- **Shell-command construction shares one escaping helper.** `src/main/ssh/shellEscape.ts` (single-quote POSIX escaping, embedded `'` → `'\''`) is imported by both `UnzipService.ts` (building `unzip`/`tar` commands) and `TerminalManager.ts` (injecting a `cd` on terminal open) — don't reintroduce a local private copy in a new call site; import this one.
- **Archive-extension recognition lives in `src/shared/archive.ts`, not hardcoded per call site.** `archiveKind()`/`isArchive()` are pure functions (no Electron/Node deps) imported by both the renderer (`fileTypes.ts`, re-exported for `FileRow.vue`'s "Extract here" gating) and the main process (`UnzipService.ts`, choosing `unzip` vs. `tar -xf`/`-xzf`/`-xjf` per kind) — this is the shared-module pattern to follow for any future renderer/main-agreed classification logic (mirrors how `contract.ts` itself is shared, but scoped to domain logic rather than IPC shapes). `.7z`/`.rar` are deliberately unrecognized (not reliably available on remote hosts) — don't add them without also designing new missing-tool error handling.
- **The Terminal lands in the site's configured remote start path, not the SSH default.** SSH's shell-channel protocol has no cwd concept at all (confirmed against `@types/ssh2`) — `TerminalManager.open()` writes a `cd <shellEscape(path)>\n` into the freshly-opened stream (after listeners are attached, so the shell's response is forwarded like any typed command) when `SessionManager.cwd(sessionId)` resolves to something other than `.`. That value is already seeded from `Site.remoteInitialPath` at connect time for the file browser's benefit — this just added a second consumer, no new plumbing. If you add another feature that needs a session's configured start path, read `SessionManager.cwd(sessionId)`, don't add new IPC plumbing for it.

Reuse it for future similar cases instead of writing a new one-off transition.
- **Do not manufacture a transition by re-keying/re-mounting `FilePane`/`BottomDock`/`TerminalView` per active site tab.** Switching between two already-connected tabs deliberately does NOT swap those components — same instances, different session data via computed getters (see the `remoteFs.store.ts`/`terminalStreams.store.ts` entries above) — forcing a remount there to animate the switch would tear down live xterm scrollback and cached fs listings that this architecture specifically preserves. The only top-level state boundary that gets a fade is the picker↔connected swap in `App.vue`.
- **CSS transitions on a collapse/expand toggle need explicit sizes on BOTH states — a `flex-1`/no-explicit-value side will not animate.** `FilePane.vue`'s Local-pane rail collapse animates `width` between `w-10` (collapsed) and `min-w-0 w-1/2` (expanded, LOCAL side only — the ternary is conditioned on `side === 'local'`, since `FilePane.vue` is instantiated twice from one template and the Remote pane must keep its own unrelated `flex-1` untouched in both of Local's states, or it won't grow to fill the space Local's rail frees up). `min-w-0 flex-1` (flex-grow-derived, no explicit `width`, computed style stays `width: auto`) was tried first and produced NO visible animation at all — CSS cannot interpolate to/from an `auto`-resolved width, only between two concrete `<length-percentage>` values (`BottomDock.vue`'s collapse/expand works the same way, now via an inline `:style="{ height: ... }"` binding — `36px` collapsed, `ui.dockHeight`px expanded — rather than the old fixed `h-9`/`h-56` Tailwind classes, still two explicit lengths). If a future pane/panel collapse needs the same treatment, give both states an explicit length — never leave one side as bare `flex-1`/`auto`.
- **A generic in-flight IPC-call counter (`ipcActivity.store.ts`), not hand-picked per-action flags, drives the title-bar busy indicator.** `api.ts`'s `invoke()` increments/decrements this store's `count` around every call (skipping a small `QUIET_CHANNELS` exclusion set: `terminal:write`/`terminal:resize` — fire per keystroke/resize — and the four window-chrome channels, semantically unrelated to "SSH/SFTP work"). `useGlobalActivity.ts`'s `isBusy` is primarily this counter (`ipcActivity.isBusy`), so any new IPC channel is automatically covered without remembering to wire it in — the composable's `connecting`/transfer-count/fs-`loading` checks exist only to pick a nicer LABEL when they're the active reason, falling back to a generic "Working…" otherwise. If you add a new high-frequency channel (fires on a timer or per-keystroke/per-frame), add it to `QUIET_CHANNELS`, don't leave it tracked.
- **A composable that returns multiple refs (e.g. `useGlobalActivity()`'s `{ isBusy, label }`) must be destructured at the call site, not accessed via the returned object's dot-notation, if used directly in a template.** `const activity = useGlobalActivity(); ...{{ activity.isBusy }}...` does NOT auto-unwrap — `activity` is a plain object, and Vue's template compiler only auto-unwraps refs that are top-level `<script setup>` bindings (or accessed through a `reactive()`-wrapped object), not nested properties reached via dot-access on a plain object. Always write `const { isBusy, label } = useGlobalActivity()` and reference `isBusy`/`label` directly in the template. Getting this wrong doesn't error at build time — it silently passes the raw `Ref` object as a prop/condition (a `v-if` on a raw ref object is always truthy, and a `:prop="someRef"` binding throws a runtime "invalid prop type" warning), so watch for that failure shape if a composable's boolean/value never seems to update in a template.
- **Recursive directory transfer is one `TransferQueue` job, not N.** `enqueueTree()` pushes a `TransferJob` with `isTree: true` into the SAME `pending`/`active` queue as single-file jobs (so the existing `MAX_CONCURRENT`/cancel machinery needs no changes), but `runTree()` then walks the whole source tree (`LocalFsService.listRecursive`/`RemoteShell.readdirRecursive`), recreates every directory at the destination (`mkdir -p`-equivalent, tolerant of already-existing dirs), and transfers its files. `runFile()` (the byte-streaming core, shared by both a solo file job and each file inside a tree job) takes an `onProgress` callback rather than broadcasting directly, so `runTree()` can aggregate all its files' bytes into ONE throttled `progress` event under the job's single `transferId` — the renderer never learns a tree transfer contains more than one file.
- **Directories and files WITHIN one tree job run up to `TREE_ITEM_CONCURRENCY` (4) at a time, not one at a time** — a Phase-3-performance-review fix: a serial walk makes wall time dominated by per-file/per-mkdir round-trip latency rather than bandwidth on anything but a very low-latency link (the review's own example: ~5,000 small files at 150ms latency was ~3-5 files/sec serial). `runConcurrent<T>(items, concurrency, worker)` is the generic bounded-worker-pool helper this uses (a shared `nextIndex` cursor + N `runNext()` loops racing via `Promise.all`) — reuse it for any future "N independent async steps, bounded fan-out, stop-on-first-failure" need rather than writing a new one. Because files now finish out of order, `runTree()`'s progress aggregation changed from one `currentFileBytes` number to an `inFlightBytes: Map<index, bytes>` (summed alongside `doneBytes` on each `emitProgress()` call) — if you touch this aggregation again, remember multiple files can be simultaneously "in flight" now, not just one. `TREE_ITEM_CONCURRENCY` is deliberately separate from `MAX_CONCURRENT` (concurrent tree/file JOBS, not files within one job) — don't conflate the two constants.
- **`RemoteShell.deleteRecursive` is exec-first (`rm -rf --`), not a per-entry SFTP walk — this was a real perf fix, not a style choice.** The original implementation `stat`ed and `readdir`ed its way down a tree, `unlink`/`rmdir`-ing one entry at a time — deleting a folder with hundreds of files was hundreds of sequential SFTP round-trips, dramatically slower than a plain `rm -rf` run over the same SSH connection. `deleteRecursive` now runs a single `rm -rf -- <shellEscape(path)>` exec (same connection this app already keeps open for Tail/Terminal/Unzip/Compress) and only falls back to the renamed-private `deleteRecursiveSftp` walk if the exec throws (no shell/exec capability at all — a restricted sftp-only chroot account) or returns non-zero. `RemoteShell.integration.test.ts` has a real-server test asserting exactly one `exec()` call fires for a nested multi-file tree, plus a test that force-fails `exec()` (via `vi.spyOn`) and confirms the SFTP fallback still deletes correctly. If you ever need to force the fallback path deliberately, that spy pattern is the way to test it — there's no public flag to select it.
- **fs stores patch `entries` in place after a mutation — they do NOT reload the whole directory.** `remove`/`rename`/`chmod` (remote only) in both `localFs.store.ts` and `remoteFs.store.ts` used to end with `await this.load()`, paying a full extra `readdir` round-trip (remote) or a full local re-list after every single delete/rename/chmod — and multi-select delete looped this once *per selected file*. They now splice/patch the in-memory `entries` array directly (`rename` re-runs `compareEntries` since a new name can change sort position; `chmod` just overwrites `entry.permissions`), and a new `removeMany(entries)` action on both stores fires all the underlying deletes concurrently via `Promise.allSettled` before patching once — entries whose delete failed stay in the listing, and `removeMany` throws a summary error naming them so the caller can surface one toast instead of silently dropping failures. `FilePane.vue`'s multi-select Delete/Backspace handler calls `removeMany`, not a `for` loop over singular `remove`. If you add a new mutating fs action, follow this patch-in-place pattern instead of reaching for `load()` — reserve `load()` for actual navigation and the explicit Ctrl+R refresh.
- **File-list sort and selection logic are pure functions, not inlined into the stores.** `renderer/src/utils/fileSort.ts`'s `compareEntries(column, direction)` and `fileSelection.ts`'s `selectOnly`/`toggleSelect`/`selectRange`/`selectAll` are imported identically by both `localFs.store.ts` (flat state) and `remoteFs.store.ts` (per-session `PerSessionFs` bucket) — each store just assigns the pure function's result onto its own state shape. `selectRange` needs a `selectAnchor` (the last plain-click or ctrl-click target) alongside `selected`; both stores carry it, reset to `null` on every fresh `load()`. Keep new sort/selection behavior in these shared modules, not duplicated per store, if a third fs-like store is ever added.
- **Host-key verification is trust-on-first-use (TOFU), pinned per `host:port` in `KnownHostsStore` (`known_hosts.json`, electron-store-backed).** `SessionManager.buildConnectConfig()` builds a `HostVerifier` (ssh2's `ConnectConfig.hostVerifier`) around the pure decision function `evaluateHostKey(knownFingerprint, presentedFingerprint, forceTrust)` in `KnownHostsStore.ts` — `'trust-new'` (nothing on file yet, or the caller force-trusts a change) writes the fingerprint and calls `verify(true)`; `'match'` calls `verify(true)` without writing; `'mismatch'` calls `verify(false)`, and the surrounding `connect()` catch block turns that into a `HOST_KEY_MISMATCH` `SshError` naming both fingerprints. Never call `verify(false)` without having stashed the expected/presented pair somewhere the catch block can read — the raw ssh2 error on a `hostVerifier` rejection is a generic, unhelpful "connection failed" otherwise.
- **A `let` reassigned only inside a closure passed to another function gets incorrectly narrowed by TypeScript at read sites *outside* that closure — use a mutable holder object (`{ value: T | null }`) instead of a bare nullable `let` whenever a callback (not straight-line code) is what sets it.** Hit this for real building the host-key-mismatch plumbing: `let hostKeyMismatch: X | null = null` reassigned only inside the `hostVerifier` callback, then read in the `catch` block via `hostKeyMismatch ? ... : ...`, type-checked to `never` in the truthy branch — TypeScript's control-flow analysis doesn't track assignments inside nested functions, so it treated the read site as if the variable could only ever be `null`. `SessionManager.ts`'s `HostKeyMismatchHolder` interface is the fix; reach for the same pattern (not a type-assertion workaround) for any future "a callback sets this, code after the callback reads it" case.
- **`SessionManager.connect()`'s real 2FA/OTP flow only forwards prompts it can't answer itself — it does NOT forward every keyboard-interactive challenge to the user.** `keyboardInteractive.ts`'s pure `partitionPrompts(prompts, password)` auto-answers anything matching `/password/i` (many servers route plain password auth through keyboard-interactive under PAM — confirmed for real against the test container) and only escalates the rest via `promptKeyboardInteractive()` (broadcasts `session:keyboard-interactive-prompt`, resolves when `session:keyboard-interactive-respond` arrives with a matching `requestId`, tracked in `SessionManager`'s `pendingKeyboardInteractive` map). `mergeAnswers()` recombines both halves back into ssh2's expected answer-array order. If you need to add auto-answer heuristics for another common prompt shape, extend `partitionPrompts` — don't special-case it inline in `SessionManager.connect()`.
- **Jump-host tunneling reuses the exact same per-hop connect machinery as the main target, called twice.** `SessionManager.connect()` calls `buildConnectConfig()`/`connectClient()` once for the jump host (if `input.jumpHost` is set) and once for the real target — each hop gets its own `HostVerifier`/`KnownHostsStore` entry (keyed by its own `host:port`), and real keyboard-interactive prompts from EITHER hop route through the same `sessionId` (harmless — it's just informational context for the renderer dialog, not a routing key; routing is by `requestId`). The target's `ConnectConfig.sock` is set to the `ClientChannel` returned by `jumpClient.forwardOut()` (`forwardThroughJump()`) — ssh2 then tunnels the whole second SSH handshake through that stream instead of opening its own TCP socket. `SessionEntry.jumpClient` is ended alongside the main `client` in `close()` and on any connect failure — a jump-host session that never cleans up its bastion hop would leak a connection on every failed/closed session.
- **`defaultAgentPath()` (`SessionManager.ts`) is the only place platform-specific ssh-agent resolution lives — don't inline `process.platform` checks elsewhere for this.** Windows defaults to the modern OpenSSH Agent's named pipe (`\\.\pipe\openssh-ssh-agent`); non-Windows defaults to `$SSH_AUTH_SOCK`, throwing `VALIDATION` if unset. A site's `agentPath` override (surfaced in `SiteFormDialog.vue`) takes precedence over this default entirely — including the literal string `pageant`, which `ssh2`'s own `createAgent()` special-cases to talk to PuTTY's Pageant instead of a named pipe/socket. This path has NOT been verified against a real running agent in this environment (no agent was available) — unlike jump-host, which was verified via a self-jump trick against the real test container.
- **`ChmodDialog.vue`'s 3×3 checkbox grid is written as explicit owner/group/other `<tr>`s, not a `v-for` over an object keyed by triplet name.** A `v-for="(label, key) in {owner:'Owner',...}"` + `tripletsObj[key]` pattern was considered and rejected — indexing a reactive object by a `v-for`-widened key fails `vue-tsc`'s strict template type-checking (the inferred key type is a plain `string`, not a literal union). Three explicit rows is more code but the only version that actually type-checks cleanly; follow this if another fixed-cardinality grid ever needs the same shape.
- **`permissions.ts`'s `parseMode`/`formatMode` are the ONLY sanctioned octal-string <-> triplet conversion — `toTechnical`/`toFriendlyLabel`'s internal `parseTriplet` stayed private on purpose.** `ChmodDialog.vue` needs round-trip (parse for the initial checkbox state, format back to a mode string to submit) where the read-only display helpers only ever needed one direction. If a future feature needs the triplets themselves (not just a formatted string), import `parseMode`/`Triplet`/`ModeTriplets` from `permissions.ts` rather than re-deriving bitmasks inline.
- **Transfer retry re-enqueues from scratch under a NEW `transferId` — it does not resume or reuse the failed job.** `transferQueue.store.ts`'s `retry()` deletes the old (`error`/`cancelled`) entry from its `items` map and calls `this.enqueue()` again with the same `sessionId`/`kind`/paths/`isDir` it cached on the original entry — this is why `TransferItemState` grew `sessionId`/`isDir` fields (previously only implicit in the `enqueue()` call's arguments, never persisted). Nothing in `TransferQueue.ts` (main process) knows about "retry" — it's purely a renderer-side "do the same enqueue call again" convenience, which is also why it needed no new IPC channel and no new real-server test (the underlying `enqueue()`/`enqueueTree()` path is already covered).
- **A site's stored secret is retained ONLY when its current `authMethod` actually uses it — switching methods clears the other, it does not just stop displaying it.** `SiteStore.ts`'s `pickSecret(wants, provided, existing)` (used by both `create()`/`update()` for the site itself and `toStoredJumpHost()` for its jump-host hop) returns `undefined` whenever `wants` is false, even if an old encrypted value exists — and `SessionManager.openFromSite()` ALSO gates `secrets.password`/`secrets.passphrase` on `site.authMethod`/`site.jumpHost.authMethod` before passing them on, as defense-in-depth for sites saved before this existed. This closes a real credential-leak: without it, switching a site from password auth to privateKey/agent left the old password ciphertext in `sites.json` forever, and `SessionManager`'s keyboard-interactive auto-answer (`keyboardInteractive.ts`'s `partitionPrompts`, matching `/password/i`) would keep silently replaying that stale password into any matching prompt regardless of the site's now-different configured auth method. If you add a new secret field tied to a specific auth method, gate it through `pickSecret` the same way rather than a bare `input.x !== undefined ? encrypt(input.x) : existing.x`.
- **Any numeric/enum-shaped IPC field that gets interpolated into a shell command must be sanitized to a concrete safe value at the point it's first accepted from IPC — never trust the TypeScript param type.** `ipc/*.ts` handlers cast `req as SomeRequest` with zero runtime validation (`envelope.ts`'s `handle()` doesn't validate shapes either) — the actual value crossing the context-bridge is whatever the renderer sent, which a compromised/buggy renderer could make anything. `TailManager.ts`'s `sanitizeHistoryLines()` (clamps to a non-negative integer, `Math.trunc(Number(value))` with a `200` fallback on non-finite) is the pattern: fix this once, in the class that owns the value and does the interpolation (`tail -n ${historyLines}`), not by trying to validate every call site that might produce that value.
- **Saved sites support a free-text `group` field (`Site.group`/`SiteInput.group`/`StoredSite.group`, all optional) for organizing the site list** — deliberately a flat string tag, not a nested folder tree (WinSCP's actual model), to keep `SessionManagerView.vue`'s grouping logic simple: `groupedSites` computed there renders a flat list with NO section headers at all if no site has a group set (so users who never touch this feature see the exact same UI as before), and only switches to per-group `<h>`-style headers (alphabetical, with an "Ungrouped" section trailing) once at least one site has one. `sites.store.ts`'s `groupNames` getter (distinct group names, sorted) feeds `SiteFormDialog.vue`'s "Group" field's `<datalist>` suggestions — a plain HTML datalist, not a Nuxt UI combobox, since this is just autocomplete convenience, not a constrained enum.
- **Site duplicate (`sites:duplicate` → `SiteStore.duplicate(id)`) clones the `StoredSite` record as-is, including its already-encrypted secret ciphertext — it does NOT decrypt-then-re-encrypt.** `safeStorage`'s encryption is OS-account-scoped (DPAPI on Windows), not per-record-keyed, so copying the ciphertext byte-for-byte onto a new `id`/`name`/timestamps is both correct and cheaper than a decrypt/re-encrypt round-trip. Follow this if you add any other "clone a secret-bearing record" feature — never round-trip through plaintext just to duplicate a row.
- **Bandwidth throttling is a virtual-schedule limiter (`TransferQueue.ts`'s `RateLimiter`), not a capped token bucket — this distinction matters, it was a real bug.** An earlier capped-bucket draft (`tokens = Math.min(limit, tokens + elapsed*limit)`) deadlocked forever on any single `acquire()` request bigger than one second's worth of budget (e.g. a 64KB stream chunk against a 10 KB/s limit), since capacity was capped at exactly `limit` and could never grow enough to satisfy the request. The shipped design instead tracks `nextAvailableAt` (a timestamp) and reserves each request's proportional duration synchronously, before awaiting anything — a request larger than one second's budget just takes proportionally longer rather than hanging. `RateLimiter.test.ts` (fake timers) covers this specifically; if you ever touch this class, keep a test for "a single request bigger than the per-second budget still resolves." The limiter is a single instance shared by ALL concurrent transfers (not per-file), applied via a `Transform` stream spliced into `runFile()`'s existing `readStream.pipe(writeStream)` chain (now `readStream.pipe(throttle).pipe(writeStream)`) — this is the pattern to extend if a future feature needs per-chunk backpressure hooked into the same pipe.
- **`archiver` v8's API is NOT the old `archiver('zip', opts)` factory function — it's `new ZipArchive(opts)`/`new TarArchive(opts)`/`new JsonArchive(opts)` (named class exports), a breaking change from earlier majors.** Confirmed against the installed version's actual `node_modules/archiver/index.js` (pure ESM, `export class ZipArchive extends Archiver`) after `@types/archiver`'s `.d.ts` had no default export to satisfy the old call style — `CompressService.ts`'s `compressLocal()` is the reference usage (`new ZipArchive({ zlib: { level: 9 } })`, then `.directory()`/`.file()`/`.pipe()`/`.finalize()` same as always). `CompressService.test.ts` includes a real-filesystem smoke test asserting the output's first 4 bytes are the ZIP local-file-header magic number (`50 4B 03 04`) — typecheck alone would NOT have caught a wrong construction call, this test would.
- **Small app-wide settings that must survive a restart but aren't a saved site live in `AppSettingsStore` (`app-settings.json`, electron-store), NOT `localStorage`.** This is the dividing line from `ui.store.ts`: `ui.store.ts`'s prefs (local-pane visibility, permissions display mode, theme) are pure renderer/`localStorage` state with no main-process involvement; `AppSettingsStore`-backed settings (open-tab restoration, bandwidth limit) need main-process code to read them anyway (`TransferQueue`'s limit, `sessions.store.ts`'s tab restore needs the site list) so they're persisted main-process-side and reached via `settings:get`/`settings:setOpenTabs`/`settings:setBandwidthLimit` IPC instead. If a new persisted setting is renderer-only and never read by main, use `ui.store.ts`'s pattern; if main process needs to read/act on it, extend `AppSettingsStore` instead.
- **Restored site tabs are deliberately NOT auto-connected.** `sessions.store.ts`'s `restoreOpenTabs()` (called once from `App.vue` at startup) recreates a picker tab (siteId/label/hostLabel set, `sessionId`/`status` still null) for every `openTabSiteIds` entry that still resolves to a saved site — it never calls `openSession`/`connectToSite` itself. This was a deliberate call, not an oversight: auto-reconnecting with saved credentials (and potentially firing a keyboard-interactive/2FA challenge) the instant the app launches, before the user has done anything, was judged more surprising than useful for a security-conscious SFTP client — mirrors how browsers lazily restore/discard tabs rather than eagerly reloading every one. `persistOpenTabs()` (writes the current siteId list) is called from `openSession()` right after a tab's `siteId` is assigned (so intent is persisted even before a connect attempt resolves) and from `closeTab()` — if you add a new place a tab's `siteId` changes, persist there too.
- **WinSCP/PuTTY session import (`SessionImporter.ts`) deliberately never imports a password, for either source.** PuTTY doesn't store one at all; WinSCP's stored password is only reversibly *obfuscated* (XOR-based, not real encryption), but re-implementing that scheme without a real WinSCP install available in this environment to validate against risked silently importing a wrong plaintext credential with no visible error — worse than asking the user to retype it. If a future session wants to add real WinSCP password decoding, it MUST be validated against an actual WinSCP-written registry entry (not just hand-authored fixtures) before shipping, precisely because a subtly-wrong decode is a silent-failure mode, not a loud one. The registry-format PARSING itself (`regQuery.ts`'s `parseRegValues`/`listRegSubkeyPaths`/`parseDword`/`decodeSessionName`) IS unit-tested against hand-authored `reg query`-style fixtures — that part doesn't need a real registry to validate, only the actual `reg.exe` invocation (`SessionImporter.ts`'s `regQuery()`, via `execFile`, never a shell — no injection risk since `key` is one argv element) is unverified against a real machine. Only one level of `Sessions` subkeys is scanned — a WinSCP session folder-group (nested subkeys) is silently not recursed into; if you add that, surface a count of skipped nested groups in `ImportSessionsDialog.vue` rather than dropping them with no signal.
- **`OperationRegistry.run()` (round 6) is the generalized `TransferEvent` pattern for non-transfer long ops — instrumentation happens at the IPC-handler level, not inside the services.** `UnzipService.extractRemote`, `CompressService.compressRemote`/`compressLocal`, and `RemoteFsService.removeRemote` stay plain, registry-free, unit-testable functions; `unzip.ipc.ts`/`archive.ipc.ts`/`fs.ipc.ts`'s handlers wrap the actual call in `OperationRegistry.getInstance().run({kind, label, sessionId, cancellable}, ({signal, reportProgress}) => ...)`. If you add a new instrumented operation, thread `signal`/`onProgress` through the SERVICE function's own signature (optional params) rather than reaching into `OperationRegistry` from inside `main/unzip`/`main/archive`/`main/fs` — keep the registry a handler-level concern. A batch of N renderer-issued single-item calls (the multi-select delete) becomes ONE main-side operation (`fs:remote:deleteMany`) rather than the renderer correlating N separate operation rows — this is the pattern for any future "select many, act on all" feature.
- **Deletes are never cancellable; extract/compress are.** `RemoteShell.deleteRecursive`'s `rm -rf` can't be meaningfully aborted mid-flight without misleading the user about what actually survived, so `delete-remote`/`delete-remote-batch` operations always report `cancellable: false` and the Activity row never renders a cancel button for them. Aborting `extract-remote`/`compress-remote` just stops the renderer from waiting — the remote `unzip`/`zip -r` process may still finish server-side (no PID-capture kill, unlike `TailManager`); `compress-local`'s cancel is a true local cancel (archiver `.abort()` + unlink the partial zip).
- **`MonitorManager` (round 6) is a buffered-`exec`-on-a-self-rescheduling-`setTimeout` poller, NOT a long-lived `execLines` stream like `TailManager`.** Each 2s tick is one self-contained `RemoteShell.exec()` round-trip over the session's existing connection — a dropped/failed tick just misses one sample, so there's no PID-capture/kill machinery, no reconnect-with-backoff, and no idle reaper (it's only running while the Monitor dock tab is mounted, started/stopped by `MonitorPanel.vue`'s lifecycle hooks, not by session connect/disconnect). If a future feature needs a genuinely continuous remote stream (not a poll), `TailManager`'s shape is still the one to copy — don't retrofit that machinery onto `MonitorManager` for its own sake.
- **`/proc` parsing lives in pure `procParse.ts`, with `MonitorManager` treating "no `/proc/stat` at all" as `unsupported`, not `error`.** A BSD/macOS/locked-down-shell target simply has no `/proc` — `MonitorManager.tick()` checks `parseProcStat(...).length === 0` on the FIRST tick specifically to distinguish "this host will never support this" (broadcast `unsupported` once, stop, no retry) from "this tick's output was transiently unparseable" (count toward `MAX_CONSECUTIVE_FAILURES`, then `error`). If you extend the sampled metrics, keep new parsers pure and side-effect-free in `procParse.ts` so they stay unit-testable without a server, following `parseMeminfo`'s MemAvailable-fallback shape for any other "modern field with an older-kernel fallback" case.
- **BottomDock's open/active-tab state moved into `useDockState.ts` (module-scoped singleton refs, the `useSettingsDialog.ts` pattern) so other components can open the dock on a specific tab.** `TitleBar.vue`'s busy indicator calls `useDockState().openDock('activity')` on click; don't reach back into `BottomDock.vue`'s own local refs for this — they no longer exist there. A dock tab that must survive `BottomDock.vue`'s own tab-switching still uses `v-show` (Terminal only, per the pre-existing convention) — Activity/Monitor are stateless-enough to remount freely since their renderer stores (`operations.store.ts`/`monitor.store.ts`) hold the actual state.
- **The dock's expanded height is drag-resizable via a `pointerdown`/`pointermove`/`pointerup` handle in `BottomDock.vue`, using `setPointerCapture` (no `window`-level listener cleanup needed) and a local `liveHeight` ref that only commits to `ui.store.ts`'s persisted `dockHeight` (one `localStorage` write) on `pointerup` — NOT on every `pointermove`.** The CSS `transition-[height]` used for the collapse/expand toggle is conditionally dropped (`dragging ? '' : 'transition-[height] ...'`) while a drag is in progress, so a real-time drag isn't fighting a 200ms easing curve. `clampDockHeight` (pure, unit-tested in `ui.store.test.ts`) is reused both at store-init (reading the persisted value) and on the window's `resize` event (re-clamping against a new viewport) — don't duplicate the bounds math inline anywhere else.
- **The Activity dock tab deliberately does NOT toast on operation completion — call sites already do.** `operations.store.ts`'s `ensureSubscription()` only merges `operation:event` into its map; `FilePane.vue`'s `onCompress`/`performExtract` and the delete actions already call `useNotify()` with richer context (filenames, counts) on success/error. Toasting again from the store would double-notify — if you add a new instrumented operation, let its existing call site own the toast, don't add one in `operations.store.ts`.
- **The Terminal's Ctrl+C/Ctrl+V handling lives in `terminalStreams.store.ts` via `term.attachCustomKeyEventHandler`, decided by the pure `terminalKeys.ts` (`terminalKeyAction`), not inline in the store or in `TerminalView.vue`.** Paste reads the clipboard through a new main-process-side `system:clipboardReadText` IPC (Electron's `clipboard.readText()`) rather than `navigator.clipboard.readText()` — the sandboxed renderer's version is permission-and-focus-dependent across Electron/OS combinations, the main-process read is unconditional. If you need another keyboard shortcut inside the terminal, extend `terminalKeyAction`'s decision table, not the handler in `terminalStreams.store.ts` directly, so it stays unit-testable without xterm.js.
- **Main-process security posture is defense-in-depth on top of the already-on `contextIsolation`/`sandbox`/`nodeIntegration:false` window flags (`main/index.ts`).** `createWindow()` also (a) denies ALL renderer permission requests via `session.defaultSession.setPermissionRequestHandler(... callback(false))` — the app needs no camera/mic/geolocation/notifications, so a blanket deny is correct, not per-permission; (b) blocks every top-level navigation with an unconditional `webContents.on('will-navigate', e => e.preventDefault())` — there's no `vue-router`, so any navigation is illegitimate and `will-navigate` never fires for the initial `loadURL`/`loadFile`; (c) restricts `setWindowOpenHandler`'s `shell.openExternal(url)` to `http:`/`https:` schemes only (parsed via `new URL`), so a link rendered from remote SSH data can't invoke an arbitrary OS handler. If you add a feature that legitimately needs a permission, an in-app navigation, or a non-http external scheme, relax the specific guard deliberately — don't remove it wholesale. IPC handlers (`ipc/envelope.ts`) deliberately do NOT validate `event.senderFrame` (single window, no `<webview>`/remote content — the guards above already prevent an untrusted frame from existing); revisit only if a `<webview>` or remote-content feature is ever added.
- **Electron Fuses are locked down in `electron-builder.yml`'s `electronFuses:` block** (`runAsNode:false`, `enableNodeOptionsEnvironmentVariable:false`, `enableNodeCliInspectArguments:false`, `enableCookieEncryption:true`, `enableEmbeddedAsarIntegrityValidation:true`, `onlyLoadAppFromAsar:true`). electron-builder 26.x applies these to the packaged binary itself (`@electron/fuses`, pulled transitively — no direct dependency). Verify a change with `npx @electron/fuses read --app dist/win-unpacked/Ferry.exe` after packaging — a mis-set `onlyLoadAppFromAsar`/integrity fuse produces an app that won't launch, invisible to typecheck/vitest, so `npm run package` + launching the unpacked exe is the only real smoke test.
- **Heavy main-process modules are lazy-loaded at their point of use, not imported at module top-level, to keep them off the app-startup path.** `SessionManager.ts` imports `Client` from `ssh2` as `import type` only and pulls the runtime class via `const { Client: SshClient } = await import('ssh2')` inside `connect()` (aliased to dodge the file-wide `Client` type name); `CompressService.ts` does `const { ZipArchive } = await import('archiver')` inside `compressLocal()`. Both `ssh2` and `archiver` are in `electron.vite.config.ts`'s `main.build.rollupOptions.external` so the `import()` stays a runtime require (verify with `grep 'import("ssh2")' out/main/index.js` after a build). This mirrors the renderer's "type-only import + dynamic import at point of use" pattern (icons/xterm/`FilePane`) — follow it for any new expensive-to-load main-process dependency that isn't needed until a specific user action.
- **`FileList.vue` is virtualized with `@tanstack/vue-virtual` (`useVirtualizer`) — rows are absolutely positioned inside a `getTotalSize()`-tall spacer, only `getVirtualItems()` mount.** It uses dynamic `measureElement` (each row wrapper gets `:ref="measureRow"` + `:data-index`), NOT a fixed `estimateSize`, because a row's height shifts with the inline rename `UInput` and the remote permissions column — measuring self-corrects instead of risking overlap/gaps. Keys are `entry.path` (via `getItemKey`) so a sort/reorder doesn't remount rows and lose hover/rename state. `FileRow.vue` is unchanged (it's the single consumer). If you add a per-row element that changes height, measurement already handles it — don't switch to a fixed size.

## Gotchas that will waste your first hour

1. **`electron-builder.yml` must be named EXACTLY that** (or `electron-builder.ts`/`.json`/etc.) — NOT `electron-builder.config.ts`. electron-builder's config loader (`read-config-file`) only searches for `electron-builder.<ext>`; `*.config.ts` is silently ignored with no error, and you get default settings (`oneClick: true`, wrong product name, no asarUnpack). We hit this for real — see `.claude/plan/ferry-implementation-plan.md` history. Also: electron-builder's `.ts`-config loader (`config-file-ts`) has a Windows bug where absolute paths with a drive letter break its cache-directory naming — prefer `.yml` over `.ts` for this config on Windows regardless.
2. **Icons will silently 404/CSP-block unless the full Lucide set is registered — AND the network-fallback path is disabled, not just deferred.** `@nuxt/ui`'s icon component falls back to fetching unrecognized icon names from Iconify's public API at runtime — blocked by our CSP (`default-src 'self'`), and Vite's static icon-bundling can't detect dynamically-bound icon names (computed props, ternaries) anyway. `main.ts` registers the full Lucide set via a dynamic `import()` deferred to `requestIdleCallback` (so cold-start isn't blocked on parsing 554KB of JSON) — but that alone isn't sufficient, since icons rendered in the first frame (before the deferred load resolves) would still hit the network-fallback path and get CSP-blocked. `main.ts` also calls `setCustomIconLoader(loader, 'lucide')` (from `@iconify/vue`) with a loader that `await`s the SAME memoized load-then-`addCollection()` promise used by the idle-scheduled call, returning `null` once it resolves — this is NOT a no-op/never-resolving loader (a loader that never settles would permanently strand any icon that hit it before the real load finished, since Iconify only re-notifies already-mounted `Icon` instances when a load attempt actually *settles*, not merely when `addCollection()` is later called on its own). If you add a different icon collection, register both halves (the deferred `addCollection` load AND a `setCustomIconLoader` gating on that same promise) the same way — don't just add a bare `addCollection()` call.
3. **`titleBarStyle: 'hidden'` alone removes ALL window chrome on Windows** — no close/minimize/maximize button, only Alt+F4 — and that's now intentional: `TitleBar.vue` draws its own buttons wired to `window:*` IPC (see Conventions). Do NOT re-add `titleBarOverlay` "to be safe" — it draws real OS caption buttons over the custom bar with no way to hide just the buttons while keeping the overlay, which is exactly what we removed it to avoid.
4. **`tsconfig.web.json`'s `include` must cover `src/renderer/auto-imports.d.ts`, not just `src/renderer/src/**/*`.** `unplugin-auto-import` (via `@nuxt/ui/vite`) writes global ambient declarations (`useToast`, `useOverlay`, etc.) to that file, one directory above the renderer source root. Component tags (`UButton`, `UModal`, ...) still resolve fine in `<template>` even without it, because Vue's template checker (`@vue/language-core`) discovers `components.d.ts`-style files through a separate mechanism — but a bare composable call in `<script setup>` (e.g. `useToast()`) goes through the real TS program and needs the ambient file in `include` to type-check. Hit this for real adding `useNotify.ts` — fixed by adding the path explicitly.
5. **OS drag-and-drop resolves a dropped file's path via `window.api.getPathForFile()` (preload → `webUtils.getPathForFile`), NOT the DOM `File.path` extension.** `File.path` was removed in Electron 32; this app now runs Electron 43, so `FilePane.vue`'s `onOsDrop` calls the preload-bridged `getPathForFile(file)` (added to the `ElectronAPI` surface in both `preload/index.ts` and `renderer/src/env.d.ts`) to get the absolute path of each dropped item. If OS-drag uploads ever silently stop (files drop but nothing enqueues), check that this bridge method still resolves a non-empty path — a future Electron may move `webUtils` again. The *outgoing* local-row drag (`system:startDrag`) is unaffected: it hands `entry.path` (already known from the fs listing) to the main-side `WebContents.startDrag`, never touching a DOM `File`.
6. **App icon is now set** — `resources/1024x1024.png` is electron-builder's `win.icon` source (auto-generates every Windows resolution from one square PNG), `resources/favicon.ico` is the `BrowserWindow`'s dev-mode-only icon (`main/index.ts`, gated on `IS_DEV` since packaged builds inherit the exe's embedded icon instead), and `src/renderer/public/favicon.ico` (referenced via a relative `<link rel="icon">` in `index.html`) covers the (mostly invisible, given the custom titlebar) renderer-side favicon.
7. `@toys\ui` (the local `@nuxt/ui` source clone) is NOT a dependency of this project and has no build output — it's reference material only (source browsing + the `nuxt-ui` skill). Never point tooling at it.
8. **File preview/download/tail affordances only apply to the remote pane, never local.** `FilePreviewDialog.vue`'s download and tail buttons are gated on `side === 'remote'` — downloading or tailing a file that's already on the local machine doesn't make sense, and `TailManager`/`tail:start` only ever operates over an SSH session. If you add a similar affordance elsewhere, gate it the same way (mirrors `FilePane.vue`'s pre-existing `showTail = side === 'remote' && connected` computed).
9. **Never call `xterm.js`'s `term.open()` more than once per `Terminal` instance.** Reparenting a live terminal onto a second element isn't a documented-safe operation, so `TerminalView.vue` guards it with a `Set` of already-attached session ids and relies on the container staying permanently in the DOM (`v-show`, see Conventions) instead. If a future refactor needs the container to actually unmount/remount, dispose the old `Terminal` and create a fresh one rather than re-opening it.
10. **Auto-update is genuinely untested scaffolding, not a working feature yet.** `AutoUpdater.ts` wires `electron-updater` against `electron-builder.yml`'s `publish: {provider: github, owner: REPLACE_WITH_GITHUB_OWNER, repo: REPLACE_WITH_GITHUB_REPO}` — those placeholders MUST be replaced with a real repo (and an actual GitHub Release published for it) before any of this does anything; until then `checkForUpdates()` fails every time (caught and silently swallowed in `AutoUpdater.ts`'s `error` handler — deliberately, since there's nothing actionable for a user to do about a missing feed). It's also a no-op in dev (`app.isPackaged` guard) — you cannot test this via `npm run dev`, only against a real packaged+published build. Code signing similarly has zero project-specific config: `electron-builder.yml` relies entirely on electron-builder auto-detecting the standard `CSC_LINK`/`CSC_KEY_PASSWORD` env vars at package time (unset in this dev environment → unsigned local builds, intentional via `forceCodeSigning: false`). Neither auto-update nor code signing has ever been exercised against reality in this environment — flag both if a future session reports either "broken", since "never verified" and "broken" look identical from here.
11. **Terminal output crosses IPC as raw `Uint8Array`, not a decoded string.** `TerminalManager` forwards each `stream.on('data', chunk: Buffer)` as `new Uint8Array(chunk)` rather than `chunk.toString()`, because a naive per-chunk UTF-8 decode can split a multi-byte character across two chunks and corrupt it; `xterm.js`'s `Terminal.write()` accepts `Uint8Array` directly so no decode step is needed in the renderer either. Keystrokes going the other way (`terminal:write`) are a plain string — that direction has no chunk-boundary problem.
12. **A real bug, not a hypothetical: `TerminalView.vue` never called `term.focus()`, so the Terminal accepted NO keyboard input (not even Ctrl+C) until the user clicked inside the xterm canvas first.** Fixed (round 6) by calling `terminalStreams.focus(sessionId)` at the end of `syncActiveTerminal()`, which already fires on every "terminal became visible" path (dock-tab click, dock expand, site-tab switch, first open) via its existing `[props.active, sessions.activeSessionId]` watcher — no new call site was needed, just the missing `.focus()`. If a future dock tab or embedded widget "doesn't respond to the keyboard at all," check for this exact class of bug (focus never given) before assuming the input pipeline itself is broken.
13. **This Electron app cannot be visually verified by opening its renderer dev server URL (`http://localhost:5173`, printed by `npm run dev`) in a plain browser tab.** The renderer depends entirely on `window.api` (the `contextBridge` whitelist from `src/preload/index.ts`), which only exists inside a real Electron `BrowserWindow` — a bare Chromium tab has no preload, so `window.api` is `undefined` and `#app` mounts empty (confirmed: `document.getElementById('app').innerHTML.length === 0`, no thrown console error). This is expected, not a regression. What CAN be checked this way: that `main`/`preload`/`renderer` all built without errors in the `electron-vite dev` log, and — since Tailwind v4's JIT scanner runs against the same source — that any new literal Tailwind class strings (e.g. `colorForFile`'s per-bucket colors) actually appear in the served `main.css`'s generated `@theme` block, which is a fast way to confirm a class string wasn't silently dropped without launching the full app. Genuine interactive verification (does the UI actually render/behave right) requires either running the packaged/dev Electron window directly (not verifiable through the Browser preview tool) or a real user with a real SSH server. A cheap "does it at least boot under this Electron" smoke test that IS runnable headlessly: launch the built app (`npx electron .` on `out/`, or the packaged `dist/win-unpacked/Ferry.exe`) for ~10s and confirm multiple `electron.exe`/`Ferry.exe` processes stay alive (a startup crash from a breaking change or a bad fuse exits immediately) — then `Stop-Process` them.
14. **`tsc --build`/`vue-tsc --build` are INCREMENTAL — after any dependency change (`npm install`, a version bump) a "clean" typecheck can report stale/phantom errors, or hide real ones, until the `.tsbuildinfo` cache is cleared.** Real example: right after the Electron 30→43 install, `npm run typecheck` reported `MonitorManager.ts` errors that did not exist in a clean full check (`find . -name '*.tsbuildinfo' -not -path './node_modules/*' -delete` then re-run → clean). If a typecheck result looks surprising immediately after touching dependencies, delete the tsbuildinfo files and re-run before believing (or debugging) it.
15. **On this Windows machine, `npm install`/`npm run package` intermittently fail with `EBUSY`/`EPERM` on file renames** (electron binary swap into `node_modules/.electron-*`, or NSIS-resource extraction into `AppData\Local\electron-builder\Cache`) — a transient AV/file-lock issue, not a config error. The fix is simply to re-run the same command: the electron postinstall/`electron-builder` both detect the stale/partial state and finish on the second attempt (packaging even logs "Detected stale extracting state ... re-extracting"). If `npm install` leaves `node_modules/electron/dist/version` missing (interrupted postinstall), run `node node_modules/electron/install.js` to finish the binary download. Before packaging, make sure no prior `Ferry.exe`/`electron.exe` from a smoke test is still running (it holds `icudtl.dat` locked → `EBUSY`).
16. **One integration test — `SessionManager.jumphost.integration.test.ts` — currently fails with `CHANNEL_OPEN_FAILURE` ("Jump host tunnel ... open failed") in this environment, and it fails identically on `main` / without any of the recent changes.** It's an environmental limitation of the local Docker SSH container (the forwarded channel is refused), NOT a code regression — verified by running the same test against unmodified `SessionManager.ts`. The other integration suites self-skip when no container is up; this one runs but can't complete the tunnel. Don't chase it as a bug introduced by a change unless it starts failing at a DIFFERENT point than `forwardThroughJump`'s `forwardOut`.
